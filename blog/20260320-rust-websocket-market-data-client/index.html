用 Rust 写一个不会断连的 WebSocket 行情客户端

⏱ 2026年3月20日
· 26 分钟阅读
#rust#trading#websocket#async#tokio

写交易系统时，WebSocket 客户端是最容易被低估的组件。看起来就是”连上去然后读消息”，实际上在生产里跑几天你就会发现各种问题：连接莫名断掉、服务端超时踢人、重连风暴把交易所的限速触发了、延迟突然飙高但程序完全没感知。

我之前用过几个现成的库，但行情场景有些特殊需求，最终还是自己实现了一套。这篇记录一下核心逻辑。

依赖版本

[dependencies]
tokio = { version = "1.50", features = ["full"] }
tokio-tungstenite = "0.29"
futures-util = "0.3"
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"

tokio-tungstenite 0.29 对应 tungstenite 0.29，这两个要版本对齐，否则编译时会有 trait 冲突。

数据结构

先把核心数据类型定清楚：

use serde::{Deserialize, Serialize};
use std::time::Duration;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Tick {
    pub symbol: String,
    pub price: f64,
    pub volume: f64,
    pub timestamp_ms: u64,  // 交易所的事件时间戳
}

#[derive(Debug, Clone)]
pub struct MarketData {
    pub tick: Tick,
    pub received_at: std::time::SystemTime,  // 本地接收时间，用于计算延迟
}

received_at 单独存是为了和 tick.timestamp_ms 做差，得到端到端延迟。后面会用到。

配置也单独抽出来：

pub struct WsClientConfig {
    pub url: String,
    pub ping_interval: Duration,
    pub reconnect_base: Duration,   // 首次重连等待时间
    pub reconnect_max: Duration,    // 重连等待上限
    pub channel_capacity: usize,
}

impl Default for WsClientConfig {
    fn default() -> Self {
        Self {
            url: "ws://localhost:8080".to_string(),
            ping_interval: Duration::from_secs(30),
            reconnect_base: Duration::from_millis(500),
            reconnect_max: Duration::from_secs(60),
            channel_capacity: 1024,
        }
    }
}

reconnect_base 设 500ms，每次失败翻倍，上限 60s。这是个常见的指数退避（exponential backoff）配置，避免在服务端宕机时发起重连风暴。

客户端主结构

use tokio::sync::broadcast;

pub struct WsMarketClient {
    config: WsClientConfig,
    tx: broadcast::Sender<MarketData>,
}

impl WsMarketClient {
    pub fn new(config: WsClientConfig) -> (Self, broadcast::Receiver<MarketData>) {
        let (tx, rx) = broadcast::channel(config.channel_capacity);
        (Self { config, tx }, rx)
    }
}

用 broadcast::channel 而不是 mpsc，是因为行情数据通常需要广播给多个消费者——策略线程、风控线程、日志线程可能同时在订阅。

重连循环

这是核心部分：

use futures_util::{SinkExt, StreamExt};
use tokio::time::sleep;
use tokio_tungstenite::{connect_async, tungstenite::Message};

impl WsMarketClient {
    pub async fn run(self) {
        let mut backoff = self.config.reconnect_base;

        loop {
            match connect_async(&self.config.url).await {
                Ok((ws_stream, _)) => {
                    println!("connected to {}", self.config.url);
                    backoff = self.config.reconnect_base; // 连接成功，重置退避

                    let (mut sink, mut stream) = ws_stream.split();
                    let ping_interval = self.config.ping_interval;
                    let tx = self.tx.clone();

                    // 心跳任务单独跑
                    let ping_task = tokio::spawn(async move {
                        loop {
                            sleep(ping_interval).await;
                            let ts = std::time::SystemTime::now()
                                .duration_since(std::time::UNIX_EPOCH)
                                .unwrap()
                                .as_millis() as u64;
                            let payload = ts.to_be_bytes().to_vec();
                            if sink.send(Message::Ping(payload.into())).await.is_err() {
                                break;
                            }
                        }
                    });

                    // 主消息循环
                    while let Some(msg) = stream.next().await {
                        match msg {
                            Ok(Message::Text(text)) => {
                                let received_at = std::time::SystemTime::now();
                                if let Ok(tick) = serde_json::from_str::<Tick>(&text) {
                                    let _ = tx.send(MarketData { tick, received_at });
                                }
                            }
                            Ok(Message::Pong(data)) => {
                                if data.len() == 8 {
                                    let sent_ms = u64::from_be_bytes(
                                        data[..8].try_into().unwrap()
                                    );
                                    let now_ms = std::time::SystemTime::now()
                                        .duration_since(std::time::UNIX_EPOCH)
                                        .unwrap()
                                        .as_millis() as u64;
                                    let rtt = now_ms.saturating_sub(sent_ms);
                                    println!("ws rtt: {}ms", rtt);
                                }
                            }
                            Ok(Message::Close(_)) => break,
                            Err(e) => {
                                eprintln!("ws error: {e}");
                                break;
                            }
                            _ => {}
                        }
                    }

                    ping_task.abort();
                    println!("disconnected, reconnecting in {:?}", backoff);
                }
                Err(e) => {
                    eprintln!("connect failed: {e}");
                }
            }

            sleep(backoff).await;
            backoff = (backoff * 2).min(self.config.reconnect_max);
        }
    }
}

几个设计决策解释一下：

心跳和消息循环分离

心跳需要写端 sink，消息接收需要读端 stream。split() 之后 sink 被移进 ping_task，stream 留在主循环。两者互不干扰。ping_task 通过 sink.send() 的返回值感知断连，主循环通过 stream.next() 的错误感知断连，任一侧断掉都会触发重连。

Ping payload 带时间戳

Ping 的 payload 塞了当前时间戳（8字节大端），Pong 原样返回。这样在处理 Pong 时就能算出来回时间（RTT）。真正的网络延迟比 RTT 小一半，但 RTT 足够用来做异常检测——如果连续几个 Pong 的 RTT > 500ms，基本说明网络有问题了。

saturating_sub 防溢出

时间差用 saturating_sub 而不是直接减，是因为本地时钟和交易所时钟可能不同步，极端情况下差值可能为负（表现为溢出）。这个处理简单但够用。

重连后 backoff 归零

backoff = self.config.reconnect_base 这一行在连接成功时执行，保证下一次断连时从头开始退避，而不是从上次的值继续。

延迟监控

行情客户端里最容易被忽略的一块是延迟监控。交易所的 timestamp_ms 是事件发生时间，本地的 received_at 是消息到达时间，两者之差就是端到端延迟（当然要求本地和交易所时钟对齐，用 NTP 的话精度一般在几毫秒以内）。

pub struct LatencyTracker {
    samples: Vec<u64>,
    window: usize,
}

impl LatencyTracker {
    pub fn new(window: usize) -> Self {
        Self {
            samples: Vec::with_capacity(window),
            window,
        }
    }

    pub fn record(&mut self, md: &MarketData) {
        let exchange_ms = md.tick.timestamp_ms;
        let local_ms = md.received_at
            .duration_since(std::time::UNIX_EPOCH)
            .unwrap()
            .as_millis() as u64;
        let latency = local_ms.saturating_sub(exchange_ms);
        self.samples.push(latency);
        if self.samples.len() > self.window {
            self.samples.remove(0);
        }
    }

    pub fn p99(&self) -> Option<u64> {
        if self.samples.is_empty() {
            return None;
        }
        let mut sorted = self.samples.clone();
        sorted.sort_unstable();
        let idx = (sorted.len() as f64 * 0.99) as usize;
        Some(sorted[idx.min(sorted.len() - 1)])
    }

    pub fn mean(&self) -> Option<f64> {
        if self.samples.is_empty() {
            return None;
        }
        Some(self.samples.iter().sum::<u64>() as f64 / self.samples.len() as f64)
    }
}

用滑动窗口存最近 N 条的延迟，然后算 p99 和均值。p99 比均值更有用，因为行情延迟的分布通常有长尾——99% 的时候是 20ms，偶尔来一个 500ms 的尖刺，均值可能完全体现不出来。

实际用法：

let mut tracker = LatencyTracker::new(1000); // 最近1000条

// 在消息处理循环里
let md = rx.recv().await?;
tracker.record(&md);

// 每分钟打一次日志
if let (Some(mean), Some(p99)) = (tracker.mean(), tracker.p99()) {
    println!("latency mean={:.1}ms p99={}ms", mean, p99);
}

完整使用示例

#[tokio::main]
async fn main() {
    let config = WsClientConfig {
        url: "wss://stream.binance.com:9443/ws/btcusdt@trade".to_string(),
        ping_interval: Duration::from_secs(30),
        reconnect_base: Duration::from_millis(500),
        reconnect_max: Duration::from_secs(60),
        channel_capacity: 1024,
    };

    let (client, mut rx) = WsMarketClient::new(config);

    // 客户端在独立任务里跑
    tokio::spawn(client.run());

    let mut tracker = LatencyTracker::new(1000);
    let mut count = 0u64;

    loop {
        match rx.recv().await {
            Ok(md) => {
                tracker.record(&md);
                count += 1;

                if count % 100 == 0 {
                    if let (Some(mean), Some(p99)) = (tracker.mean(), tracker.p99()) {
                        println!(
                            "[{}] price={} latency mean={:.1}ms p99={}ms",
                            md.tick.symbol,
                            md.tick.price,
                            mean,
                            p99
                        );
                    }
                }
            }
            Err(broadcast::error::RecvError::Lagged(n)) => {
                eprintln!("consumer lagged by {} messages", n);
            }
            Err(broadcast::error::RecvError::Closed) => break,
        }
    }
}

注意 RecvError::Lagged 这个错误——当消费者处理太慢，broadcast channel 的 ring buffer 满了之后会丢弃旧消息，receiver 下一次 recv 会拿到 Lagged(n)，n 是丢了多少条。这个要处理，不然线程死锁或者策略用了过期数据。

踩过的坑

tokio-tungstenite 的版本对齐问题

用 tokio-tungstenite 时，Message 类型来自 tungstenite（它的依赖）。如果项目里同时直接依赖了 tungstenite，但版本和 tokio-tungstenite 内部用的不一样，你会碰到”类型相同但 trait 不匹配”的编译错误。解决方法是要么完全不直接依赖 tungstenite，要么明确在 Cargo.toml 里指定和 tokio-tungstenite 一致的版本。

sink 在 spawn 之后的生命周期

ws_stream.split() 拿到的 sink 和 stream，sink 被 move 进 ping_task，stream 留主循环。Rust 的所有权系统在这里帮了忙——不可能同时在两个地方使用 sink，编译期就拦住了。但要注意：如果主循环退出后没有 ping_task.abort()，ping_task 会继续持有 sink，直到下一次发送失败才停下来，这段时间里连接处于”zombie”状态。

时钟偏差

本地时间和交易所时间可能相差几十毫秒，NTP 也只是尽力同步。所以延迟数据里可能有负值（本地时间早于交易所时间）。用 saturating_sub 会把负值变成 0，这对延迟统计来说是合理的处理。
