Rust 自定义内存分配器实战：手写 Bump Allocator 与 TypedArena

⏱ 2026年3月16日
· 31 分钟阅读
#rust#memory#allocator#arena#systems-programming

写 Rust 写久了，习惯了 Box::new 和 Vec::push 背后的全局分配器，很少去想每次分配到底在干什么。直到前段时间我在做一个 AST parser，profiling 发现 40% 的时间花在 malloc/free 上——每个 AST 节点都是一次堆分配，几十万个节点下来，全局分配器扛不住了。

翻了一下 rustc 的源码，发现编译器自己大量使用 TypedArena 来分配 AST 和 HIR 节点。游戏引擎里也常见类似的做法：per-frame allocator 每帧开始时重置，分配只需要移动一个指针。这类分配器统称 bump allocator 或 arena allocator，原理简单，效果很好。

我花了一个下午自己写了两个版本，这篇文章记录整个实现过程。

全局分配器为什么慢

malloc 要维护一个空闲链表（或类似结构），每次分配需要在链表里找到合适大小的块，free 的时候还要合并相邻空闲块。这些操作在高频分配场景下开销不小：

  • 查找空闲块：遍历链表，O(n) 最坏情况
  • 碎片管理：合并、拆分空闲块
  • 线程安全：多线程环境下需要加锁（jemalloc 用 thread-local cache 缓解，但仍有开销）

如果你的使用场景满足一个条件——对象的生命周期是批量的，比如一次编译过程中的所有 AST 节点、一帧游戏中的所有临时对象——就可以用一种极简策略：只分配不释放，最后一把全部回收。

这就是 bump allocator。

Bump Allocator：一个指针走天下

结构极其简单：一块预分配的连续内存，一个偏移量指针。分配就是把指针往前移，释放只有 reset 一条路。

pub struct BumpAllocator {
    buf: *mut u8,
    capacity: usize,
    offset: Cell<usize>,
}

buf 是底层的连续内存块，offset 记录当前分配到哪里了。用 Cell<usize> 而不是 usize，是因为 alloc 方法签名是 &self 而不是 &mut self——这样调用方可以同时持有多个分配出来的引用，不会跟借用检查器打架。

初始化

impl BumpAllocator {
    pub fn new(capacity: usize) -> Self {
        let layout = Layout::from_size_align(capacity, 16)
            .expect("invalid layout");
        let buf = unsafe { alloc::alloc(layout) };
        if buf.is_null() {
            alloc::handle_alloc_error(layout);
        }
        Self {
            buf,
            capacity,
            offset: Cell::new(0),
        }
    }
}

初始化的时候从全局分配器拿一大块内存。对齐到 16 字节是个保守选择，覆盖了所有常见类型的对齐要求。

分配

分配是整个 bump allocator 的核心，也是它快的原因：

pub fn alloc<T>(&self, val: T) -> &T {
    let layout = Layout::new::<T>();
    let current = self.offset.get();

    // 对齐
    let aligned = (current + layout.align() - 1) & !(layout.align() - 1);
    let new_offset = aligned + layout.size();

    if new_offset > self.capacity {
        panic!(
            "BumpAllocator: out of memory (need {}, have {})",
            new_offset, self.capacity
        );
    }

    self.offset.set(new_offset);

    unsafe {
        let ptr = self.buf.add(aligned) as *mut T;
        ptr::write(ptr, val);
        &*ptr
    }
}

这里有个细节：对齐。如果前一次分配了一个 u8（1 字节），下一次分配 u64（对齐要求 8 字节），中间需要跳过 7 个字节的 padding。(current + align - 1) & !(align - 1) 是标准的向上对齐公式，位运算一步到位。

分配操作的时间复杂度：O(1)。没有链表遍历，没有锁竞争，就是一次加法和一次位运算。

重置

pub fn reset(&self) {
    self.offset.set(0);
}

一行搞定。把偏移量归零，所有之前分配的内存逻辑上全部回收。注意这里不会调用任何对象的析构函数——如果你分配了 String 或者 Vec 这种拥有堆内存的类型，reset 之后那些堆内存就泄漏了。

这不是 bug，是设计取舍。Bump allocator 适合的场景就是 Copy 类型或者生命周期明确由外部管理的类型。如果需要析构，得用 arena。

Drop

impl Drop for BumpAllocator {
    fn drop(&mut self) {
        let layout = Layout::from_size_align(self.capacity, 16)
            .expect("invalid layout");
        unsafe {
            alloc::dealloc(self.buf, layout);
        }
    }
}

整块底层内存在 allocator 被 drop 时归还给全局分配器。

100 个 u64 占了 800 字节（100 × 8），没有一个字节的额外开销。全局分配器做不到这点——malloc 的实现（如 glibc 的 ptmalloc）每次分配都有 bookkeeping 开销，通常是 16-32 字节的 chunk header。

TypedArena：支持析构的 Arena 分配器

Bump allocator 的问题是不调用析构函数。如果分配的对象里面有 String、Vec 或者文件句柄，reset 会导致资源泄漏。

TypedArena 的思路：记住分配了什么，drop 的时候逐个调用析构函数。代价是只能分配同一种类型（类型化），但对于 AST 节点这种场景刚好合适——rustc 就是给每种节点类型开一个独立的 arena。

数据结构

pub struct TypedArena<T> {
    chunk_capacity: usize,
    chunks: RefCell<Vec<ArenaChunk<T>>>,
    len: Cell<usize>,
}

struct ArenaChunk<T> {
    storage: Vec<mem::MaybeUninit<T>>,
    len: usize,
}

跟 bump allocator 的一块连续内存不同，TypedArena 用多个 chunk 组成一个链。当前 chunk 满了就新开一个。MaybeUninit<T> 让我们可以预分配空间但不初始化，等实际 alloc 的时候再写入。

Chunk 的实现

impl<T> ArenaChunk<T> {
    fn new(capacity: usize) -> Self {
        let mut storage = Vec::with_capacity(capacity);
        unsafe {
            storage.set_len(capacity);
        }
        Self { storage, len: 0 }
    }

    fn is_full(&self) -> bool {
        self.len >= self.storage.len()
    }

    fn push(&mut self, val: T) -> &T {
        assert!(!self.is_full(), "chunk is full");
        let slot = &mut self.storage[self.len];
        slot.write(val);
        self.len += 1;
        unsafe { slot.assume_init_ref() }
    }
}

set_len(capacity) 这行看着吓人但是安全的——MaybeUninit<T> 不要求初始化，不会有未初始化内存被读取的问题。push 的时候用 MaybeUninit::write 写入值，然后用 assume_init_ref 拿到引用。

Chunk 的 Drop

这是 TypedArena 比 bump allocator 多出来的关键部分：

impl<T> Drop for ArenaChunk<T> {
    fn drop(&mut self) {
        for i in 0..self.len {
            unsafe {
                ptr::drop_in_place(self.storage[i].as_mut_ptr());
            }
        }
        unsafe {
            self.storage.set_len(0);
        }
    }
}

逐个遍历已初始化的元素调用它们的 drop。最后把 storage 的 len 设为 0——严格来说 MaybeUninit<T> 没有实现 Drop，Vec drop 时不会对元素调用析构。但显式清零 len 是个防御性习惯，万一将来改了内部结构不至于出问题。

分配与自动扩容

pub fn alloc(&self, val: T) -> &T {
    let mut chunks = self.chunks.borrow_mut();
    let last = chunks.last_mut().expect("no chunks");

    if last.is_full() {
        let new_cap = (self.chunk_capacity * 2).min(4096);
        chunks.push(ArenaChunk::new(new_cap));
        let new_last = chunks.last_mut().unwrap();
        self.len.set(self.len.get() + 1);
        let ptr = new_last.push(val) as *const T;
        return unsafe { &*ptr };
    }

    self.len.set(self.len.get() + 1);
    let ptr = last.push(val) as *const T;
    unsafe { &*ptr }
}

chunk 满了就新开一个，容量翻倍（上限 4096）。通过裸指针返回引用来绕开 RefCell 的借用检查——这在这里是安全的，因为 arena 只追加不移动，已有 chunk 里的元素地址不会变。

写个测试验证析构

不放心的话，可以用一个 drop counter 来验证：

#[test]
fn test_drop_called() {
    thread_local! {
        static DROP_COUNT: Cell<u32> = const { Cell::new(0) };
    }

    struct Tracked;
    impl Drop for Tracked {
        fn drop(&mut self) {
            DROP_COUNT.with(|c| c.set(c.get() + 1));
        }
    }

    DROP_COUNT.with(|c| c.set(0));

    {
        let arena: TypedArena<Tracked> = TypedArena::new(4);
        for _ in 0..10 {
            arena.alloc(Tracked);
        }
    } // arena drop 在这里

    let count = DROP_COUNT.with(|c| c.get());
    assert_eq!(count, 10, "all 10 objects should be dropped");
}

10 个对象分配，跨了 2 个 chunk（第一个容量 4，第二个容量 8，装了 4 + 6），arena drop 之后 counter 精确到 10。

两种分配器的对比

Bump AllocatorTypedArena
分配速度O(1)，极快O(1)，接近 bump
支持析构不支持支持
类型约束任意类型混合分配单一类型
内存回收reset() 批量回收drop 时自动回收
适用场景Copy 类型、临时计算AST 节点、编译器 IR
内存开销几乎为零每个 chunk 有 Vec 的 overhead

实际用在哪里

rustc 编译器就是 TypedArena 的重度用户。rustc_arena crate 是编译器的基础设施之一，每种 AST/HIR/MIR 节点类型对应一个 arena。整个编译过程中的节点分配全走 arena，编译结束统一释放。

游戏引擎通常用 bump allocator 做 per-frame allocation。每帧开始 reset，帧内的所有临时分配（粒子、碰撞检测结果、渲染命令）都走 bump，帧结束不用逐个释放。

ECS 框架（如 bevy 的底层）用 arena 存储组件数据，保证同类组件在内存中连续排列，对 CPU cache 友好。

src/bump.rs

use std::alloc::{self, Layout};
use std::cell::Cell;
use std::ptr;

pub struct BumpAllocator {
    buf: *mut u8,
    capacity: usize,
    offset: Cell<usize>,
}

impl BumpAllocator {
    pub fn new(capacity: usize) -> Self {
        let layout = Layout::from_size_align(capacity, 16).expect("invalid layout");
        let buf = unsafe { alloc::alloc(layout) };
        if buf.is_null() {
            alloc::handle_alloc_error(layout);
        }
        Self {
            buf,
            capacity,
            offset: Cell::new(0),
        }
    }

    pub fn alloc<T>(&self, val: T) -> &T {
        let layout = Layout::new::<T>();
        let current = self.offset.get();

        let aligned = (current + layout.align() - 1) & !(layout.align() - 1);
        let new_offset = aligned + layout.size();

        if new_offset > self.capacity {
            panic!(
                "BumpAllocator: out of memory (need {}, have {})",
                new_offset, self.capacity
            );
        }

        self.offset.set(new_offset);

        unsafe {
            let ptr = self.buf.add(aligned) as *mut T;
            ptr::write(ptr, val);
            &*ptr
        }
    }

    pub fn reset(&self) {
        self.offset.set(0);
    }

    pub fn used(&self) -> usize {
        self.offset.get()
    }

    pub fn capacity(&self) -> usize {
        self.capacity
    }
}

impl Drop for BumpAllocator {
    fn drop(&mut self) {
        let layout = Layout::from_size_align(self.capacity, 16).expect("invalid layout");
        unsafe {
            alloc::dealloc(self.buf, layout);
        }
    }
}

src/arena.rs

use std::cell::{Cell, RefCell};
use std::mem;
use std::ptr;

pub struct TypedArena<T> {
    chunk_capacity: usize,
    chunks: RefCell<Vec<ArenaChunk<T>>>,
    len: Cell<usize>,
}

struct ArenaChunk<T> {
    storage: Vec<mem::MaybeUninit<T>>,
    len: usize,
}

impl<T> ArenaChunk<T> {
    fn new(capacity: usize) -> Self {
        let mut storage = Vec::with_capacity(capacity);
        unsafe {
            storage.set_len(capacity);
        }
        Self { storage, len: 0 }
    }

    fn is_full(&self) -> bool {
        self.len >= self.storage.len()
    }

    fn push(&mut self, val: T) -> &T {
        assert!(!self.is_full(), "chunk is full");
        let slot = &mut self.storage[self.len];
        slot.write(val);
        self.len += 1;
        unsafe { slot.assume_init_ref() }
    }
}

impl<T> Drop for ArenaChunk<T> {
    fn drop(&mut self) {
        for i in 0..self.len {
            unsafe {
                ptr::drop_in_place(self.storage[i].as_mut_ptr());
            }
        }
        unsafe {
            self.storage.set_len(0);
        }
    }
}

impl<T> TypedArena<T> {
    pub fn new(chunk_capacity: usize) -> Self {
        assert!(chunk_capacity > 0, "chunk capacity must be > 0");
        let first_chunk = ArenaChunk::new(chunk_capacity);
        Self {
            chunk_capacity,
            chunks: RefCell::new(vec![first_chunk]),
            len: Cell::new(0),
        }
    }

    pub fn alloc(&self, val: T) -> &T {
        let mut chunks = self.chunks.borrow_mut();
        let last = chunks.last_mut().expect("no chunks");

        if last.is_full() {
            let new_cap = (self.chunk_capacity * 2).min(4096);
            chunks.push(ArenaChunk::new(new_cap));
            let new_last = chunks.last_mut().unwrap();
            self.len.set(self.len.get() + 1);
            let ptr = new_last.push(val) as *const T;
            return unsafe { &*ptr };
        }

        self.len.set(self.len.get() + 1);
        let ptr = last.push(val) as *const T;
        unsafe { &*ptr }
    }

    pub fn len(&self) -> usize {
        self.len.get()
    }

    pub fn chunks(&self) -> usize {
        self.chunks.borrow().len()
    }
}
