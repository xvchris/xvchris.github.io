交易系统里的对象池：用 Rust 实现零分配的 Tick 处理管线

⏱ 2026年3月31日
· 33 分钟阅读
#rust#object-pool#trading#performance#zero-allocation

写交易系统写久了会养成一个条件反射：看到 Box::new 出现在热路径上就浑身不舒服。

前几天在优化一条行情处理管线时，perf 告诉我一个我其实早该猜到的事情——malloc/free 的调用次数和 tick 数据量完全成正比。每收一条 tick，分配一个 TickEvent，策略消费完就扔掉，下一条 tick 再分配一个新的。高峰期每秒几十万条 tick，也就意味着每秒几十万次堆分配和释放。

这种模式不是不能工作，jemalloc 在连续分配相同大小的小对象时效率已经很高了。但”能工作”和”延迟稳定”是两码事。allocator 内部维护 size class、thread cache 这些机制，在绝大多数情况下开销很小，但偶尔会触发跨线程的 cache 回收、slab 合并之类的操作，这些抖动在 p99 上特别明显。

对象池是个老办法了，C++ 圈子里几十年前就在用。Rust 里做这个事情反而要想清楚一些所有权问题。

最简的 SPSC 对象池

我的场景很简单：单线程，一条行情处理线程里收 tick → 解析 → 送进策略 → 回收。没有跨线程共享，不需要 Send/Sync，不需要 Arc。

核心想法：预先分配一个定长数组，维护一个 freelist（空闲索引栈），acquire 就 pop 一个索引出来，release 就 push 回去。

use std::cell::UnsafeCell;

pub struct ObjectPool<T> {
    storage: Vec<UnsafeCell<T>>,
    free_indices: Vec<usize>,
}

impl<T: Default> ObjectPool<T> {
    pub fn new(capacity: usize) -> Self {
        let mut storage = Vec::with_capacity(capacity);
        let mut free_indices = Vec::with_capacity(capacity);
        for i in 0..capacity {
            storage.push(UnsafeCell::new(T::default()));
            free_indices.push(capacity - 1 - i);
        }
        Self { storage, free_indices }
    }

    #[inline]
    pub fn acquire(&mut self) -> Option<(usize, &mut T)> {
        let idx = self.free_indices.pop()?;
        let obj = unsafe { &mut *self.storage[idx].get() };
        Some((idx, obj))
    }

    #[inline]
    pub fn release(&mut self, idx: usize) {
        debug_assert!(idx < self.storage.len());
        self.free_indices.push(idx);
    }

    pub fn available(&self) -> usize {
        self.free_indices.len()
    }
}

几个设计选择说一下：

为什么用 UnsafeCell？ acquire 返回 &mut T，但 storage 是 Vec 持有的，直接从 Vec<T> 里拿 &mut T 会跟 &self 冲突。用 UnsafeCell 把内部可变性显式标出来。因为 acquire 拿的是 &mut self，同一时刻只有一个调用者，所以 unsafe 这里是 sound 的。

为什么 freelist 用 Vec<usize> 而不是链表？ 栈操作（push/pop 尾部）在 Vec 上是 O(1) 且 cache 友好的。用侵入式链表的做法在 C 里很常见（把 next 指针嵌到对象头部），但在 Rust 里侵入式数据结构写起来很别扭，而且 Vec<usize> 的 4096 个 usize 也就 32KB，完全在 L1 cache 里。

为什么返回 (usize, &mut T) 而不是某种 RAII guard？ 在热路径上我不想要额外的 Drop 开销，哪怕是一个空 Drop。调用者自己管理 release，反正这段代码只在行情处理管线内部用，不会到处传递。debug_assert 在开发期间帮我抓 double-free 的 bug。

拿来处理 Tick 数据

#[derive(Default, Clone)]
pub struct TickEvent {
    pub symbol_id: u32,
    pub timestamp_ns: u64,
    pub bid: f64,
    pub ask: f64,
    pub bid_size: f64,
    pub ask_size: f64,
    pub flags: u8,
}

impl TickEvent {
    #[inline]
    pub fn reset(&mut self) {
        self.symbol_id = 0;
        self.timestamp_ns = 0;
        self.bid = 0.0;
        self.ask = 0.0;
        self.bid_size = 0.0;
        self.ask_size = 0.0;
        self.flags = 0;
    }
}

一条典型的处理管线是这样的：

fn process_tick_batch(pool: &mut ObjectPool<TickEvent>, batch_size: usize) -> u64 {
    let mut checksum: u64 = 0;
    let mut held: Vec<usize> = Vec::with_capacity(batch_size);

    for i in 0..batch_size {
        if let Some((idx, tick)) = pool.acquire() {
            tick.symbol_id = (i % 256) as u32;
            tick.timestamp_ns = i as u64 * 1000;
            tick.bid = 100.0 + (i as f64) * 0.01;
            tick.ask = tick.bid + 0.02;
            tick.bid_size = 10.0;
            tick.ask_size = 12.0;
            tick.flags = 1;

            checksum = checksum.wrapping_add(tick.timestamp_ns);
            checksum ^= tick.symbol_id as u64;

            held.push(idx);
        }
    }

    for idx in held {
        pool.release(idx);
    }
    checksum
}

这里 held 这个 Vec 你可能会觉得不对劲——这不是又分配了吗？没错。在真实代码里，这个 Vec 会被提到外层做成一个复用的 buffer，或者直接用固定大小的数组。这里为了展示方便没做这步。

Benchmark: 对象池 vs Box 逐个分配

空谈性能没意义，跑一下看看实际数据。

测试环境：

  • Mac Studio, Apple M2 Max, 32 GB
  • macOS 26.3.1
  • rustc 1.94.0 (4a4ef493e 2026-03-02)
  • criterion 0.5.1

设了五组对比：

use criterion::{black_box, criterion_group, criterion_main, Criterion};

fn bench_individual_alloc(c: &mut Criterion) {
    let batch = 1000;

    // 对象池: acquire → 填充 → release，逐个来回
    c.bench_function("pool_acquire_release_1000", |b| {
        let mut pool = ObjectPool::<TickEvent>::new(4096);
        b.iter(|| {
            for i in 0..batch {
                let (idx, tick) = pool.acquire().unwrap();
                fill_tick(tick, i);
                black_box(&*tick);
                pool.release(idx);
            }
        });
    });

    // Box::new 逐个分配再 drop
    c.bench_function("box_new_drop_1000", |b| {
        b.iter(|| {
            for i in 0..batch {
                let mut tick = Box::new(TickEvent::default());
                fill_tick(&mut tick, i);
                black_box(&*tick);
            }
        });
    });

    // 栈上复用同一个变量（理论下界）
    c.bench_function("stack_reuse_1000", |b| {
        b.iter(|| {
            let mut tick = TickEvent::default();
            for i in 0..batch {
                fill_tick(&mut tick, i);
                black_box(&tick);
            }
        });
    });

    // 对象池批量持有再批量归还
    c.bench_function("pool_batch_hold_1000", |b| {
        let mut pool = ObjectPool::<TickEvent>::new(4096);
        b.iter(|| {
            let mut held = Vec::with_capacity(batch);
            for i in 0..batch {
                let (idx, tick) = pool.acquire().unwrap();
                fill_tick(tick, i);
                black_box(&*tick);
                held.push(idx);
            }
            for idx in held {
                pool.release(idx);
            }
        });
    });

    // Box 批量分配持有再 drop
    c.bench_function("box_batch_hold_1000", |b| {
        b.iter(|| {
            let mut ticks: Vec<Box<TickEvent>> = Vec::with_capacity(batch);
            for i in 0..batch {
                let mut tick = Box::new(TickEvent::default());
                fill_tick(&mut tick, i);
                ticks.push(tick);
            }
            for tick in &ticks {
                black_box(&**tick);
            }
        });
    });
}

结果：

场景1000 条 tick每条 tick
pool_acquire_release1.97 µs~2.0 ns
box_new_drop15.75 µs~15.8 ns
stack_reuse (下界)294 ns~0.3 ns
pool_batch_hold2.24 µs~2.2 ns
box_batch_hold18.43 µs~18.4 ns

几点观察：

  1. 对象池比 Box 分配快了约 8 倍。每条 tick 2 ns vs 16 ns，差距来自绕过了 malloc/free 的开销。

  2. 栈复用是理论下界，294 ns 处理 1000 条 tick，基本就是纯赋值和 black_box 的开销。对象池跟这个差距不大（1.97 µs vs 0.29 µs），多出来的部分是 freelist 的 push/pop 和一次指针间接寻址。

  3. 批量持有的场景（pool_batch_hold vs box_batch_hold）差距更大，因为 Box 方案里每个对象的地址都不连续，cache line 利用率差。对象池的 storage 是连续内存，顺序 acquire 时大概率是顺序访问。

  4. 这些数字是在几乎无竞争的理想条件下测的。真实系统里 allocator 还要应对多线程 contention、内存碎片等问题，差距可能更大。

池耗尽怎么办

上面的实现在 acquire 时如果池空了就返回 None，调用者负责处理。这个行为是刻意的——在交易系统里，如果你的对象池耗尽了，说明要么池太小，要么你有泄漏（acquire 了没 release）。两种情况都应该被明确感知到，而不是默默 fallback 到堆分配。

如果你觉得 fallback 是可以接受的，改动也很简单：

#[inline]
pub fn acquire_or_create(&mut self) -> (usize, &mut T) {
    if let Some(idx) = self.free_indices.pop() {
        let obj = unsafe { &mut *self.storage[idx].get() };
        (idx, obj)
    } else {
        let idx = self.storage.len();
        self.storage.push(UnsafeCell::new(T::default()));
        let obj = unsafe { &mut *self.storage[idx].get() };
        (idx, obj)
    }
}

但我不推荐这么做。扩容意味着 Vec::push 可能触发 realloc，这会导致之前通过 acquire 拿到的所有 &mut T 引用全部失效。在 Rust 里这编不过（生命周期约束会拦住你），但如果你用了 raw pointer 版本就可能踩到 UB。

一个折中方案是在初始化时就把池开到足够大，然后用 metrics 监控 available() 的水位线。我在实际用的时候，池大小设成峰值 tick 速率 × 管线深度 × 2，基本没耗尽过。

跟 typed-arena 的区别

有人会问：这跟 typed-arena（或者标准库 nightly 的 TypedArena）有什么区别？

区别在于对象的生命周期模型不同：

  • Arena：对象分配出去就不还了，所有对象在 arena 销毁时一起释放。适合 AST 节点、编译器 IR 这种树/图结构。
  • 对象池：对象可以被逐个回收复用。适合生命周期短且循环使用的场景，像我们这里的 tick 处理。

我之前写过 Bump Allocator 和 TypedArena，那篇的场景是批处理——一次性分配一堆东西，处理完整个释放。今天这个场景不一样，需要在流式处理中反复 acquire/release 同一批对象。

生产中用到的几个小技巧

1. 预热

初始化后先跑一轮空的 acquire/release 循环，确保所有内存页都已被 fault in。避免第一批真实数据进来时触发 page fault 导致延迟尖刺。

fn warmup(pool: &mut ObjectPool<TickEvent>) {
    let cap = pool.available();
    let mut indices = Vec::with_capacity(cap);
    for _ in 0..cap {
        let (idx, _) = pool.acquire().unwrap();
        indices.push(idx);
    }
    for idx in indices {
        pool.release(idx);
    }
}

2. 水位监控

fn check_pool_health(pool: &ObjectPool<TickEvent>, total: usize) {
    let used = total - pool.available();
    let utilization = used as f64 / total as f64;
    if utilization > 0.8 {
        eprintln!(
            "WARN: pool utilization {:.1}% ({}/{})",
            utilization * 100.0, used, total
        );
    }
}

超过 80% 就报警，给你留出扩容的窗口。

3. 池大小的选择

别拍脑袋定。在测试环境里开一个 counter，记录峰值同时在池外的对象数量，然后乘以 2 作为生产配置。我一般是先开一个很大的池（比如 65536），跑几天之后看峰值水位，再缩到合适的大小。

fn process_tick_batch(pool: &mut ObjectPool<TickEvent>, batch_size: usize) -> u64 {
    let mut checksum: u64 = 0;

    for i in 0..batch_size {
        if let Some((idx, tick)) = pool.acquire() {
            tick.symbol_id = (i % 256) as u32;
            tick.timestamp_ns = i as u64 * 1000;
            tick.bid = 100.0 + (i as f64) * 0.01;
            tick.ask = tick.bid + 0.02;
            tick.bid_size = 10.0;
            tick.ask_size = 12.0;
            tick.flags = 1;

            checksum = checksum.wrapping_add(tick.timestamp_ns);

fn main() {
    let pool_size = 4096;
    let batch = 1000;
    let mut pool = ObjectPool::<TickEvent>::new(pool_size);

    // 预热
    for _ in 0..10 {
        let _ = process_tick_batch(&mut pool, batch);
    }

    let start = std::time::Instant::now();
    let iterations = 100_000;
    let mut total_checksum: u64 = 0;
    for _ in 0..iterations {
        total_checksum = total_checksum.wrapping_add(
            process_tick_batch(&mut pool, batch)
        );
    }
    let elapsed = start.elapsed();

    let total_ticks = iterations as u64 * batch as u64;
    let ns_per_tick = elapsed.as_nanos() as f64 / total_ticks as f64;

    println!("处理 {} 个 tick，耗时 {:.2?}", total_ticks, elapsed);
    println!("每个 tick 平均 {:.1} ns", ns_per_tick);
    println!("checksum: {}", total_checksum);
    println!("池剩余: {}", pool.available());
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_pool_acquire_release() {
        let mut pool = ObjectPool::<TickEvent>::new(8);
        assert_eq!(pool.available(), 8);

        let (idx, tick) = pool.acquire().unwrap();
        tick.bid = 42.0;
        assert_eq!(pool.available(), 7);

        pool.release(idx);
        assert_eq!(pool.available(), 8);
    }

    #[test]
    fn test_pool_exhaustion() {
        let mut pool = ObjectPool::<TickEvent>::new(2);
        let (i1, _) = pool.acquire().unwrap();
        let (i2, _) = pool.acquire().unwrap();
        assert!(pool.acquire().is_none());

        pool.release(i1);
        assert_eq!(pool.available(), 1);
        pool.release(i2);
        assert_eq!(pool.available(), 2);
    }

    #[test]
    fn test_reuse_same_memory() {
        let mut pool = ObjectPool::<TickEvent>::new(4);
        let (idx1, tick) = pool.acquire().unwrap();
        let ptr1 = tick as *const TickEvent;
        pool.release(idx1);

        let (idx2, tick) = pool.acquire().unwrap();
        let ptr2 = tick as *const TickEvent;
        assert_eq!(idx1, idx2);
        assert_eq!(ptr1, ptr2);
    }
}
