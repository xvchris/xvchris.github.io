Rust 过程宏实战：手写一个 derive(Builder)

⏱ 2026年3月15日
· 43 分钟阅读
#rust#proc-macro#metaprogramming#builder-pattern

上周在写一个交易策略的配置模块，结构体有 15 个字段，手写 builder 的时候感觉自己在干体力活——每多一个字段就多一套 setter、一个 Option 包装、一段 build() 里的 unwrap 逻辑。

derive_builder 这个 crate 当然能解决问题，但我一直对过程宏的运作机制停留在”知道它解析 TokenStream”的层面，实际从没动手写过。正好趁这个机会，自己实现一个够用的 derive(Builder)，把 syn 和 quote 的用法彻底搞清楚。

过程宏的三种形态

Rust 的过程宏分三类：

  • Derive 宏：#[derive(MyTrait)]，最常见，给结构体/枚举自动生成 impl 块
  • Attribute 宏：#[my_attr]，可以修改被标注的代码
  • Function-like 宏：my_macro!(...)，看着像声明宏，但能做更复杂的事

它们都有一个硬性约束：必须放在独立的 crate 里，Cargo.toml 里标记 proc-macro = true。这是因为过程宏在编译期运行，编译器需要先把这个 crate 编译成动态库，然后在编译用户代码时加载执行。

本质上就是一个函数：吃进去 TokenStream，吐出来 TokenStream。中间用 syn 把 token 流解析成 AST 方便操作，用 quote 把 AST 拼回 token 流。

目标功能

我要支持这几种场景：

  1. 必填字段 — build() 时如果没设就返回 Err
  2. Option<T> 字段 — 自动识别为可选，不设就是 None
  3. #[builder(default = "expr")] — 自定义默认值
  4. #[builder(each = "name")] — Vec 类型字段可以逐项添加

build() 返回 Result<T, String>，没什么花哨的。

宏的入口

use proc_macro::TokenStream;
use quote::{format_ident, quote};
use syn::{
    parse_macro_input, Data, DeriveInput, Fields, GenericArgument, Lit, PathArguments, Type,
};

#[proc_macro_derive(Builder, attributes(builder))]
pub fn derive_builder(input: TokenStream) -> TokenStream {
    let input = parse_macro_input!(input as DeriveInput);
    match impl_builder(&input) {
        Ok(tokens) => tokens.into(),
        Err(err) => err.to_compile_error().into(),
    }
}

attributes(builder) 告诉编译器我们的 derive 宏会用到 #[builder(...)] 这个辅助属性，不要把它当作未知属性报错。

impl_builder 返回 syn::Result，这样解析出错时能生成一个编译错误，IDE 里直接标红到对应位置，比 panic! 体验好得多。

核心实现

整个流程分四步：解析字段、判断类型、生成 builder 结构体、生成方法。

fn impl_builder(input: &DeriveInput) -> syn::Result<proc_macro2::TokenStream> {
    let name = &input.ident;
    let builder_name = format_ident!("{}Builder", name);
    let vis = &input.vis;

    let fields = match &input.data {
        Data::Struct(data) => match &data.fields {
            Fields::Named(fields) => &fields.named,
            _ => {
                return Err(syn::Error::new_spanned(
                    name,
                    "Builder only supports structs with named fields",
                ))
            }
        },
        _ => {
            return Err(syn::Error::new_spanned(
                name,
                "Builder only supports structs",
            ))
        }
    };

format_ident!("{}Builder", name) 是 quote 提供的宏，把标识符拼接起来。如果原结构体叫 Config，生成的 builder 就叫 ConfigBuilder。

接下来遍历每个字段，解析 #[builder(...)] 属性：

    let mut builder_fields = Vec::new();
    let mut builder_defaults = Vec::new();
    let mut setter_methods = Vec::new();
    let mut build_assignments = Vec::new();

    for field in fields {
        let field_name = field.ident.as_ref().unwrap();
        let field_ty = &field.ty;

        let mut default_value: Option<syn::Expr> = None;
        let mut each_name: Option<String> = None;

        for attr in &field.attrs {
            if !attr.path().is_ident("builder") {
                continue;
            }
            attr.parse_nested_meta(|meta| {
                if meta.path.is_ident("default") {
                    let value = meta.value()?;
                    let lit: Lit = value.parse()?;
                    match &lit {
                        Lit::Str(s) => {
                            let expr: syn::Expr = syn::parse_str(&s.value())?;
                            default_value = Some(expr);
                        }
                        _ => {
                            return Err(meta.error("expected string literal for default value"));
                        }
                    }
                    Ok(())
                } else if meta.path.is_ident("each") {
                    let value = meta.value()?;
                    let lit: Lit = value.parse()?;
                    match &lit {
                        Lit::Str(s) => {
                            each_name = Some(s.value());
                        }
                        _ => {
                            return Err(meta.error("expected string literal for each name"));
                        }
                    }
                    Ok(())
                } else {
                    Err(meta.error("unsupported builder attribute"))
                }
            })?;
        }

parse_nested_meta 是 syn 2.x 的推荐方式。早期版本要手动 parse::<Meta>() 然后层层匹配，写起来又臭又长。这个闭包风格清爽很多，每个分支处理一个 key，不认识的直接报错。

我一开始还 use syn::Meta，后来发现根本没用到——clippy 抓到了这个 unused import，删掉就好。这种残留物在重构属性解析方式后特别容易留下来。

字段类型判断与代码生成

这是最有意思的部分。四种情况，每种的 builder 字段类型和 setter 逻辑都不一样：

        let is_option = is_option_type(field_ty);
        let is_vec = is_vec_type(field_ty);

        if is_vec && each_name.is_some() {
            // Vec + each: builder 里直接存 Vec<T>，初始化为空
            builder_fields.push(quote! { #field_name: #field_ty });
            builder_defaults.push(quote! { #field_name: Vec::new() });
        } else {
            // 其他情况都包一层 Option
            builder_fields.push(quote! { #field_name: std::option::Option<#field_ty> });
            if let Some(ref default_expr) = default_value {
                builder_defaults.push(quote! { #field_name: Some(#default_expr) });
            } else {
                builder_defaults.push(quote! { #field_name: None });
            }
        }

Option<T> 字段在 builder 里变成 Option<Option<T>>，这是个容易忽略的细节。外层 Option 表示”用户有没有调用 setter”，内层 Option 是字段本身的类型。如果用户没调过 setter，build() 时外层是 None，展开后给原始字段赋 None；如果调过，外层是 Some(Some(val))，展开后是 Some(val)。

        // setter 生成
        if let Some(ref singular) = each_name {
            let singular_ident = format_ident!("{}", singular);
            let inner_ty = extract_inner_type(field_ty).unwrap_or(field_ty);

            setter_methods.push(quote! {
                pub fn #singular_ident(&mut self, val: #inner_ty) -> &mut Self {
                    self.#field_name.push(val);
                    self
                }

                pub fn #field_name(&mut self, val: #field_ty) -> &mut Self {
                    self.#field_name = val;
                    self
                }
            });
        } else if is_option {
            let inner_ty = extract_inner_type(field_ty).unwrap_or(field_ty);
            setter_methods.push(quote! {
                pub fn #field_name(&mut self, val: #inner_ty) -> &mut Self {
                    self.#field_name = Some(Some(val));
                    self
                }
            });
        } else {
            setter_methods.push(quote! {
                pub fn #field_name(&mut self, val: #field_ty) -> &mut Self {
                    self.#field_name = Some(val);
                    self
                }
            });
        }

each 模式同时生成两个 setter：一个单数形式（逐项 push），一个复数形式（整个 Vec 替换）。这样用起来灵活。

build() 方法里的赋值逻辑对应三种情况：

        if is_vec && each_name.is_some() {
            build_assignments.push(quote! {
                #field_name: self.#field_name.clone()
            });
        } else if is_option {
            build_assignments.push(quote! {
                #field_name: self.#field_name.clone().unwrap_or(None)
            });
        } else {
            let err_msg = format!("missing required field: {}", field_name);
            build_assignments.push(quote! {
                #field_name: self.#field_name.clone().ok_or_else(|| #err_msg.to_string())?
            });
        }

必填字段没设就返回错误信息，告诉你哪个字段漏了。clone() 是因为 build() 接受 &self，允许多次调用——代价是一次克隆，但对于配置类结构体这完全可以接受。

最终拼装

把前面收集的片段用 quote! 拼到一起：

    let expanded = quote! {
        #vis struct #builder_name {
            #(#builder_fields,)*
        }

        impl #name {
            pub fn builder() -> #builder_name {
                #builder_name {
                    #(#builder_defaults,)*
                }
            }
        }

        impl #builder_name {
            #(#setter_methods)*

            pub fn build(&self) -> std::result::Result<#name, String> {
                Ok(#name {
                    #(#build_assignments,)*
                })
            }
        }
    };

    Ok(expanded)
}

#(...).* 是 quote 的重复语法，类似声明宏里的 $(...)*，把 Vec 里的每个元素展开。

辅助函数

类型判断就是看最后一个路径段的标识符是不是 Option / Vec，提取泛型内参就是取 AngleBracketed 的第一个参数：

fn is_option_type(ty: &Type) -> bool {
    extract_type_ident(ty)
        .map(|ident| ident == "Option")
        .unwrap_or(false)
}

fn is_vec_type(ty: &Type) -> bool {
    extract_type_ident(ty)
        .map(|ident| ident == "Vec")
        .unwrap_or(false)
}

fn extract_type_ident(ty: &Type) -> Option<String> {
    if let Type::Path(type_path) = ty {
        if let Some(segment) = type_path.path.segments.last() {
            return Some(segment.ident.to_string());
        }
    }
    None
}

fn extract_inner_type(ty: &Type) -> Option<&Type> {
    if let Type::Path(type_path) = ty {
        if let Some(segment) = type_path.path.segments.last() {
            if let PathArguments::AngleBracketed(args) = &segment.arguments {
                if let Some(GenericArgument::Type(inner)) = args.args.first() {
                    return Some(inner);
                }
            }
        }
    }
    None
}

这种简单的字符串比较对 std::option::Option 这种全路径写法会失效，但实际代码里几乎没人这么写，够用了。

测试

测试覆盖了所有场景：必填字段缺失报错、Option 字段的有/无、默认值的使用和覆盖、each 的逐项添加和批量设置，以及把所有功能混合在一起的综合测试。

#[cfg(test)]
mod tests {
    use derive_builder_macro::Builder;

    #[derive(Builder, Debug, PartialEq, Clone)]
    struct Command {
        executable: String,
        args: Vec<String>,
        current_dir: String,
    }

    #[test]
    fn test_basic_required_fields() {
        let cmd = Command::builder()
            .executable("cargo".to_string())
            .args(vec!["test".to_string()])
            .current_dir("/tmp".to_string())
            .build()
            .unwrap();

        assert_eq!(cmd.executable, "cargo");
        assert_eq!(cmd.args, vec!["test".to_string()]);
        assert_eq!(cmd.current_dir, "/tmp");
    }

    #[test]
    fn test_missing_required_field() {
        let result = Command::builder()
            .executable("cargo".to_string())
            .current_dir("/tmp".to_string())
            .build();

        assert!(result.is_err());
        let err = result.unwrap_err();
        assert!(err.contains("args"), "Error should mention 'args', got: {}", err);
    }

    #[derive(Builder, Debug, PartialEq, Clone)]
    struct Config {
        name: String,
        debug: Option<bool>,
        log_file: Option<String>,
    }

    #[test]
    fn test_option_fields_set() {
        let cfg = Config::builder()
            .name("myapp".to_string())
            .debug(true)
            .log_file("/var/log/app.log".to_string())
            .build()
            .unwrap();

        assert_eq!(cfg.debug, Some(true));
        assert_eq!(cfg.log_file, Some("/var/log/app.log".to_string()));
    }

    #[test]
    fn test_option_fields_unset() {
        let cfg = Config::builder().name("myapp".to_string()).build().unwrap();
        assert_eq!(cfg.debug, None);
        assert_eq!(cfg.log_file, None);
    }

    #[derive(Builder, Debug, PartialEq, Clone)]
    struct Server {
        host: String,
        #[builder(default = "8080")]
        port: u16,
        #[builder(default = "4")]
        workers: usize,
    }

    #[test]
    fn test_default_values_used() {
        let srv = Server::builder()
            .host("localhost".to_string())
            .build()
            .unwrap();

        assert_eq!(srv.port, 8080);
        assert_eq!(srv.workers, 4);
    }

    #[test]
    fn test_default_values_overridden() {
        let srv = Server::builder()
            .host("0.0.0.0".to_string())
            .port(3000)
            .workers(8)
            .build()
            .unwrap();

        assert_eq!(srv.port, 3000);
        assert_eq!(srv.workers, 8);
    }

    #[derive(Builder, Debug, PartialEq, Clone)]
    struct Pipeline {
        name: String,
        #[builder(each = "step")]
        steps: Vec<String>,
        #[builder(each = "env_var")]
        env_vars: Vec<String>,
    }

    #[test]
    fn test_each_add_one_at_a_time() {
        let pipeline = Pipeline::builder()
            .name("deploy".to_string())
            .step("build".to_string())
            .step("test".to_string())
            .step("deploy".to_string())
            .env_var("CI=true".to_string())
            .build()
            .unwrap();

        assert_eq!(pipeline.steps, vec!["build", "test", "deploy"]);
        assert_eq!(pipeline.env_vars, vec!["CI=true"]);
    }

    #[test]
    fn test_each_empty_vec() {
        let pipeline = Pipeline::builder()
            .name("empty".to_string())
            .build()
            .unwrap();
        assert!(pipeline.steps.is_empty());
    }

    #[test]
    fn test_each_bulk_setter() {
        let pipeline = Pipeline::builder()
            .name("bulk".to_string())
            .steps(vec!["a".to_string(), "b".to_string()])
            .build()
            .unwrap();
        assert_eq!(pipeline.steps, vec!["a", "b"]);
    }

    #[derive(Builder, Debug, PartialEq, Clone)]
    struct Request {
        url: String,
        #[builder(default = "\"GET\".to_string()")]
        method: String,
        #[builder(each = "header")]
        headers: Vec<String>,
        body: Option<String>,
        #[builder(default = "30")]
        timeout_secs: u64,
    }

    #[test]
    fn test_combined_features() {
        let req = Request::builder()
            .url("https://example.com".to_string())
            .header("Content-Type: application/json".to_string())
            .header("Authorization: Bearer token".to_string())
            .body("{}".to_string())
            .build()
            .unwrap();

        assert_eq!(req.method, "GET");
        assert_eq!(req.headers.len(), 2);
        assert_eq!(req.body, Some("{}".to_string()));
        assert_eq!(req.timeout_secs, 30);
    }

    #[test]
    fn test_combined_override_defaults() {
        let req = Request::builder()
            .url("https://example.com/api".to_string())
            .method("POST".to_string())
            .timeout_secs(60)
            .build()
            .unwrap();

        assert_eq!(req.method, "POST");
        assert_eq!(req.timeout_secs, 60);
        assert!(req.headers.is_empty());
        assert_eq!(req.body, None);
    }

    #[test]
    fn test_all_required_missing() {
        let result = Command::builder().build();
        assert!(result.is_err());
    }
}

调试过程宏的几个技巧

过程宏出错时，编译器给的报错信息经常让人摸不着头脑。几个实用的调试手段：

cargo expand：这是最重要的工具。安装 cargo-expand 后，cargo expand 可以看到宏展开后的完整代码，展开结果有语法错误一眼就能看出来。

eprintln! 大法：在过程宏代码里直接 eprintln!("{:#?}", input)，编译时会把 AST 打到 stderr。粗糙但有效。

syn::Error::new_spanned：比 panic! 好用太多。panic! 只会告诉你”过程宏炸了”，new_spanned 能把错误定位到具体的 token 上，IDE 里红线直接画在出问题的代码上。

局限性

这个实现是教学性质的，跟生产级的 derive_builder 比还差不少：

  • 不支持泛型结构体
  • 不支持 where 子句
  • 类型判断只看最后一段标识符，std::option::Option<T> 会识别失败
  • setter 参数没做 Into<T> 转换
  • 没有 #[builder(setter(into))] 之类的细粒度控制

但作为理解过程宏工作原理的练习，这些都不是重点。真正在项目里用，直接上 derive_builder crate 就好。
