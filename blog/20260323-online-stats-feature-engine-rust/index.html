行情特征计算的 O(1) 问题：Welford 在线算法和滚动方差的 Rust 实现

⏱ 2026年3月23日
· 19 分钟阅读
#rust#trading#statistics#quantitative

做行情特征的时候有个问题我想了挺久：计算滑动窗口的均值和标准差，到底要不要保存窗口内所有数据？

最直白的实现是维护一个 VecDeque<f64>，每次重新算。20 个 tick 的窗口无所谓，但如果你有几十个交易品种、每个品种跑多个时间尺度的特征、每秒几千个 tick 进来，这个 O(n) 的计算开销会叠出问题。

我在项目里换成 Welford 在线算法之后，CPU 占用明显下来了。这篇把具体实现整理出来。

为什么不能用普通的方差公式

教科书上的方差公式是：

variance = (Σ(xi - mean)²) / (n - 1)

这要求你先算出均值，再扫一遍求偏差平方和，或者等价地保存 Σxi² 和 Σxi。后者看起来是 O(1)，但有个数值稳定性问题——对于接近的大数做差会丢失精度：

// 这种写法数值上不稳定
let var = (sum_sq - sum * sum / n) / (n - 1);
// 当所有 xi 都在 1000000 附近时，sum_sq 和 sum*sum/n 都是巨大的数
// 相减后精度损失严重

Welford 在 1962 年发表了一个更稳定的单次扫描算法，每次更新只需要两步：

delta = x - mean_prev
mean_new = mean_prev + delta / count
m2_new = m2_prev + delta * (x - mean_new)

最终方差就是 m2 / (count - 1)。整个过程不需要存任何历史数据，数值误差也比两次扫描小得多。

Rust 实现很直接：

#[derive(Debug, Default, Clone)]
pub struct WelfordStat {
    count: u64,
    mean: f64,
    m2: f64,
}

impl WelfordStat {
    pub fn update(&mut self, x: f64) {
        self.count += 1;
        let delta = x - self.mean;
        self.mean += delta / self.count as f64;
        let delta2 = x - self.mean;
        self.m2 += delta * delta2;
    }

    pub fn mean(&self) -> Option<f64> {
        if self.count == 0 { None } else { Some(self.mean) }
    }

    pub fn variance(&self) -> Option<f64> {
        if self.count < 2 { None } else { Some(self.m2 / (self.count - 1) as f64) }
    }

    pub fn std_dev(&self) -> Option<f64> {
        self.variance().map(f64::sqrt)
    }
}

注意 delta 和 delta2 是两个不同的量：delta 是相对旧均值的偏差，delta2 是相对新均值的偏差。这个细节是算法数值稳定性的关键，漏了就退化成不稳定版本。

滑动窗口的问题：撤出旧值

Welford 在全量统计时很好用，但交易里通常要的是”最近 N 个 tick 的统计量”，不是从头到现在的全量统计。

这就需要在加入新值的同时，把窗口最旧的值”撤出”。幸运的是，Welford 可以反向操作。设 x 是要移除的旧值，n 是移除前的样本数：

mean_new = (mean_old * n - x) / (n - 1)
m2_new = m2_old - (x - mean_old) * (x - mean_new)

这个推导是正向更新的代数逆运算，细节可以在 Knuth《计算机程序设计艺术》Vol 2 里找到。

fn remove_value(&mut self, x: f64) {
    if self.count <= 1 {
        self.count = 0;
        self.mean = 0.0;
        self.m2 = 0.0;
        return;
    }
    let old_mean = self.mean;
    self.mean = (old_mean * self.count as f64 - x) / (self.count - 1) as f64;
    self.m2 -= (x - old_mean) * (x - self.mean);
    if self.m2 < 0.0 {
        self.m2 = 0.0;  // 浮点误差保护
    }
    self.count -= 1;
}

最后那个 if self.m2 < 0.0 是个实际踩坑。理论上 m2 不可能是负数，但在接近相等的序列里（比如价格波动极小时），浮点误差会把它搓成 -1e-15 之类的小负数，然后 sqrt(-1e-15) 就 NaN 了。加个保护好过 debug 半天。

完整的滚动窗口结构：

#[derive(Debug, Clone)]
pub struct RollingWelford {
    window: VecDeque<f64>,
    capacity: usize,
    mean: f64,
    m2: f64,
    count: usize,
}

impl RollingWelford {
    pub fn new(window_size: usize) -> Self {
        assert!(window_size >= 2, "window size must be >= 2");
        Self {
            window: VecDeque::with_capacity(window_size),
            capacity: window_size,
            mean: 0.0,
            m2: 0.0,
            count: 0,
        }
    }

    pub fn push(&mut self, x: f64) {
        if self.window.len() == self.capacity {
            let old = self.window.pop_front().unwrap();
            self.remove_value(old);
        }
        self.window.push_back(x);
        self.add_value(x);
    }

    pub fn z_score(&self, x: f64) -> Option<f64> {
        let mean = self.mean()?;
        let std = self.std_dev()?;
        if std == 0.0 { return None; }
        Some((x - mean) / std)
    }
    
    // ... add_value / remove_value 同上
}

VecDeque 这里主要是为了 O(1) 的 pop_front，普通 Vec 做 shift 是 O(n)。

EMA：更轻的选择

滑动窗口需要存 N 个历史值（就算不重新算均值，也得记着要撤出谁）。EMA 更激进：完全不存历史，只保留一个标量。

EMA_new = EMA_prev + α * (x - EMA_prev)

其中 α = 2 / (N + 1) 对应传统的 EMA(N) 写法。

pub struct Ema {
    alpha: f64,
    value: Option<f64>,
}

impl Ema {
    pub fn new(period: usize) -> Self {
        Self {
            alpha: 2.0 / (period as f64 + 1.0),
            value: None,
        }
    }

    pub fn update(&mut self, x: f64) -> f64 {
        let v = match self.value {
            None => x,             // 第一个值直接初始化
            Some(prev) => prev + self.alpha * (x - prev),
        };
        self.value = Some(v);
        v
    }
}

EMA 和 Welford 不是替代关系，是不同的需求：

  • Welford：需要真实的窗口均值和方差（统计意义上严格）
  • EMA：需要一个平滑的趋势线，可以接受指数衰减的近似

我的策略里两个都用，Welford 算 Z-score，EMA 判断趋势方向。

组装成特征管道

把这几个结构组合起来的时候，我想了一下接口设计。最简单的方式是一个 FeatureEngine 结构体，内部持有各种统计器，每次收到 tick 就批量更新：

pub struct FeatureEngine {
    welford: WelfordStat,       // 全量统计（了解整体分布）
    rolling_20: RollingWelford, // 近 20 tick 统计
    ema_5: Ema,
    ema_20: Ema,
}

impl FeatureEngine {
    pub fn process(&mut self, price: f64) -> Features {
        self.welford.update(price);
        self.rolling_20.push(price);
        let ema5 = self.ema_5.update(price);
        let ema20 = self.ema_20.update(price);

        Features {
            price,
            rolling_mean: self.rolling_20.mean(),
            rolling_std: self.rolling_20.std_dev(),
            z_score: self.rolling_20.z_score(price),
            ema5,
            ema20,
            ema_diff: ema5 - ema20,
        }
    }
}

ema_diff 是 EMA5 - EMA20 的差值，正数表示短期在长期上方，常用来判断动量方向。

用一段模拟的单调上涨价格序列（从 100.0 到 107.0，每步 +0.5）跑一下，输出大概是这样：

tick= 19 price=106.5 z_score=+1.336 ema_diff=+0.4821 rolling_std=2.0494
tick= 20 price=107.0 z_score=+1.501 ema_diff=+0.6547 rolling_std=2.0494

z_score 接近 1.5 说明当前价格在窗口内偏高，配合 ema_diff 正值，是个比较典型的短期上涨状态。

一点精度问题

在实际行情里，价格往往集中在一个区间内（比如 SOL 在 150 到 155 之间震荡），窗口方差很小。这时候 m2 的值也很小，浮点精度尤其重要。

我做过一个简单的精度对比，用 [1000001.0, 1000002.0, 1000003.0] 这种大数序列：

  • 朴素公式（Σx² 减法）：误差约 1e-8 量级
  • Welford：误差约 1e-15 量级

对于均值在百位数、方差在个位数的行情数据，差距没有这么夸张，但确实有。

测试策略

几个我觉得必须有的测试：

#[test]
fn test_welford_mean_variance() {
    let mut stat = WelfordStat::new();
    for i in 1..=5 {
        stat.update(i as f64);
    }
    assert!((stat.mean().unwrap() - 3.0).abs() < 1e-10);
    assert!((stat.variance().unwrap() - 2.5).abs() < 1e-10);
}

#[test]
fn test_rolling_welford_eviction() {
    let mut rolling = RollingWelford::new(3);
    rolling.push(1.0);
    rolling.push(2.0);
    rolling.push(3.0);
    assert!((rolling.mean().unwrap() - 2.0).abs() < 1e-10);
    
    rolling.push(4.0); // 窗口变成 [2,3,4]
    assert!((rolling.mean().unwrap() - 3.0).abs() < 1e-10);
}

#[test]
fn test_ema_convergence() {
    let mut ema = Ema::new(5);
    for _ in 0..50 {
        ema.update(100.0);
    }
    assert!((ema.value().unwrap() - 100.0).abs() < 1e-8);
}

eviction 测试特别重要——光测加入新值很容易写对，撤出旧值的逻辑才是容易出 bug 的地方。

小结

这套实现没什么”高级”的地方，核心是 Welford 算法 + VecDeque ring buffer。

实际项目里你可能还需要：

  • 多个窗口长度同时维护（比如 5/10/20/60 tick 同时跑）
  • 跨品种的特征归一化
  • 特征值的 NaN 传播处理（窗口未满时很多值是 None）

这些都是工程细节，但 Welford + EMA 是地基，搞清楚了再往上堆东西会省很多力气。
