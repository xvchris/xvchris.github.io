交易执行算法的 Rust 实现：TWAP、VWAP 与 Almgren-Chriss 最优执行

⏱ 2026年3月14日
· 27 分钟阅读
#rust#quantitative-finance#trading#execution#algorithms

一笔大单怎么下？这个问题听起来简单，但做过实盘的都知道，一次性市价单砸进去，冲击成本能吃掉你大半的预期收益。执行算法（Execution Algorithm）就是解决这个问题的：把一笔大单拆成若干小单，按照某种策略分散到时间轴上执行，在市场冲击和时机风险之间找到平衡点。

这篇文章用 Rust 实现三种执行算法：TWAP、VWAP，以及基于 Almgren-Chriss 模型的最优执行。

问题背景

假设你要买入 10,000 股某标的，当前盘中流动性不算深。如果一口气市价买入，可能会把价格顶上去——你买的越多，后面的成本越高。这就是临时市场冲击（temporary market impact）。

除了冲击，还有时机风险（timing risk）：你越是分散到后面执行，价格随机游走的方差就越大，最终成交均价可能远离你决策时的参考价。

好的执行算法要在这两者之间做权衡：冲击成本和时机风险是一对矛盾，你不能同时完全消除它们。

先搭基础类型

use chrono::{DateTime, Duration, Utc};
use rand::rngs::StdRng;
use rand::{Rng, SeedableRng};

#[derive(Debug, Clone)]
pub struct OrderSlice {
    pub quantity: f64,
    pub scheduled_time: DateTime<Utc>,
}

#[derive(Debug)]
pub struct ExecutionStats {
    pub average_price: f64,
    pub total_cost: f64,
    pub slippage: f64,
    pub slippage_bps: f64,
}

OrderSlice 表示一个子单，ExecutionStats 是执行完成后的统计结果。

TWAP：最朴素的起点

TWAP（Time-Weighted Average Price）的逻辑极简：把总量均分成 N 份，每隔固定时间发一份。

pub struct TwapExecutor {
    pub total_quantity: f64,
    pub num_slices: usize,
    pub start_time: DateTime<Utc>,
    pub end_time: DateTime<Utc>,
}

impl TwapExecutor {
    pub fn schedule(&self) -> Vec<OrderSlice> {
        let qty_per_slice = self.total_quantity / self.num_slices as f64;
        let total_secs = (self.end_time - self.start_time).num_seconds();
        let interval_secs = total_secs / self.num_slices as i64;

        (0..self.num_slices)
            .map(|i| OrderSlice {
                quantity: qty_per_slice,
                scheduled_time: self.start_time
                    + Duration::seconds(interval_secs * i as i64),
            })
            .collect()
    }
}

10,000 股、12 个时间桶，每桶 833 股，间隔 30 分钟。简单，可预期，也是很多场景里的基准线。

TWAP 的问题：不管市场交量怎么分布，它都雷打不动地均分。开盘和收盘的流动性通常比午盘好得多，TWAP 在午盘跟深度好的时段用一样的量，显然没有充分利用市场结构。

VWAP：跟着市场的节奏走

VWAP（Volume-Weighted Average Price）的目标是让你的成交均价接近当天的成交量加权均价。实现思路是：根据历史交量分布（通常是当天各时段的预期成交量占比），按比例分配子单量。

A 股和美股的盘中交量通常呈 U 形——开盘和收盘量大，午盘量小。

pub struct VwapExecutor {
    pub total_quantity: f64,
    pub start_time: DateTime<Utc>,
    pub end_time: DateTime<Utc>,
    pub volume_profile: Vec<f64>, // 归一化，各桶占比之和为 1.0
}

impl VwapExecutor {
    pub fn schedule(&self) -> Vec<OrderSlice> {
        let n = self.volume_profile.len();
        let total_secs = (self.end_time - self.start_time).num_seconds();
        let interval_secs = total_secs / n as i64;

        self.volume_profile
            .iter()
            .enumerate()
            .map(|(i, &vol_frac)| OrderSlice {
                quantity: self.total_quantity * vol_frac,
                scheduled_time: self.start_time
                    + Duration::seconds(interval_secs * i as i64),
            })
            .collect()
    }

    /// 典型 U 形盘中交量曲线
    pub fn typical_u_shaped_profile(n_buckets: usize) -> Vec<f64> {
        let raw: Vec<f64> = (0..n_buckets)
            .map(|i| {
                let t = i as f64 / (n_buckets - 1) as f64;
                (1.0 - t).powi(2) * 2.0 + t.powi(2) * 2.0 + 0.5
            })
            .collect();
        let sum: f64 = raw.iter().sum();
        raw.iter().map(|&v| v / sum).collect()
    }
}

用这个 U 形 profile，12 个桶的分布大概是这样：

期 01: qty=1100  (11.0%)
期 02: qty=955   (9.5%)
...
期 06: qty=664   (6.6%)   ← 午盘低谷
期 07: qty=664   (6.6%)
...
期 12: qty=1100  (11.0%)

开盘和收盘时段量更大，贴合市场实际流动性。这样在高流动性时段多执行，实际产生的市场冲击会小一些——毕竟市场在消化你的单子的同时，也有更多其他人的交易在抵消你的冲击。

VWAP 的局限在于依赖历史交量 profile 的准确性。如果当天市场结构发生变化（比如突发消息导致午盘量暴增），预设的 profile 就不准了。

Almgren-Chriss：从数学角度看最优执行

Almgren 和 Chriss 在 2001 年发表的论文《Optimal Execution of Portfolio Transactions》给出了一个数学框架，把执行问题转化成优化问题：

最小化：E[执行成本] + λ × Var[执行成本]

其中 λ 是风险厌恶系数，控制你愿意为减少方差（即时机风险）多付多少成本。

模型假设：

  • 价格服从算术布朗运动，波动率为 σ
  • 临时市场冲击与成交量线性相关，系数为 η
  • 存在永久冲击（每笔成交会永久影响价格），但在最优路径推导中主要考虑临时冲击

连续时间的最优持仓路径为：

x(t) = X × sinh(κ(T-t)) / sinh(κT)

其中 κ = sqrt(λσ²/η)，X 是总量，T 是总执行期数。

这个 sinh 函数的解很直观：

  • λ 很小（不在乎风险）→ κ 趋近 0 → sinh 近似线性 → 退化为 TWAP
  • λ 很大（极度规避风险）→ κ 很大 → 前期持仓下降快 → 前置执行
pub struct AlmgrenChrissExecutor {
    pub total_quantity: f64,
    pub num_periods: usize,
    pub risk_aversion: f64,   // lambda
    pub market_impact: f64,   // eta
    pub volatility: f64,      // sigma
    pub start_time: DateTime<Utc>,
    pub period_duration: Duration,
}

impl AlmgrenChrissExecutor {
    pub fn optimal_trajectory(&self) -> Vec<f64> {
        let n = self.num_periods;
        let x0 = self.total_quantity;
        let kappa_sq = self.risk_aversion * self.volatility * self.volatility
            / self.market_impact;
        let kappa = kappa_sq.sqrt();
        let t_total = n as f64;

        let holdings: Vec<f64> = (0..=n)
            .map(|j| {
                let time_remaining = t_total - j as f64;
                if time_remaining <= 0.0 {
                    return 0.0;
                }
                let denom = (kappa * t_total).sinh();
                if denom < 1e-12 {
                    x0 * time_remaining / t_total
                } else {
                    x0 * (kappa * time_remaining).sinh() / denom
                }
            })
            .collect();

        // 每期成交量 = 相邻持仓差（telescoping sum 保证总量守恒）
        (0..n).map(|j| holdings[j] - holdings[j + 1]).collect()
    }

    pub fn schedule(&self) -> Vec<OrderSlice> {
        self.optimal_trajectory()
            .into_iter()
            .enumerate()
            .map(|(i, qty)| OrderSlice {
                quantity: qty,
                scheduled_time: self.start_time
                    + self.period_duration * i as i32,
            })
            .collect()
    }
}

holdings[j] - holdings[j+1] 这个相邻差是望远镜求和（telescoping sum），自然保证了所有子单加起来正好等于总量，不需要额外的修正步骤。

模拟市场，看看三种算法的差异

用一个简单的线性冲击模型模拟执行过程：

pub struct MarketSimulator {
    pub initial_price: f64,
    pub volatility: f64,      // 每期价格随机波动
    pub impact_per_unit: f64, // 每单位成交的线性冲击
    pub rng: StdRng,
}

impl MarketSimulator {
    pub fn new(initial_price: f64, volatility: f64, impact_per_unit: f64, seed: u64) -> Self {
        Self {
            initial_price,
            volatility,
            impact_per_unit,
            rng: StdRng::seed_from_u64(seed),
        }
    }

    pub fn simulate_execution(&mut self, slices: &[OrderSlice]) -> ExecutionStats {
        let mut current_price = self.initial_price;
        let mut total_cost = 0.0;
        let mut total_qty = 0.0;

        for slice in slices {
            let noise: f64 = self.rng.gen_range(-1.0..1.0);
            current_price += noise * self.volatility;
            current_price = current_price.max(0.01);

            let impact = self.impact_per_unit * slice.quantity;
            let execution_price = current_price + impact;

            total_cost += execution_price * slice.quantity;
            total_qty += slice.quantity;

            // 永久冲击：一半留在价格里
            current_price += impact * 0.5;
        }

        let avg_price = total_cost / total_qty;
        let slippage = avg_price - self.initial_price;
        ExecutionStats {
            average_price: avg_price,
            total_cost,
            slippage,
            slippage_bps: slippage / self.initial_price * 10_000.0,
        }
    }
}

用 seed=42、初始价格 100.0、波动率 0.05、冲击系数 0.001 跑一遍：

TWAP   均价=103.1088  滑点=310.88 bps
VWAP   均价=103.1305  滑点=313.05 bps
A-C    均价=107.4444  滑点=744.44 bps

A-C（λ=1.0）的滑点远高于 TWAP 和 VWAP，这其实完全符合预期。λ=1.0 时 kappa = √(λσ²/η) = √(1.0 × 0.0025 / 0.001) ≈ 1.58，kappa×T ≈ 19——这意味着算法极度前置执行，第一期就打出总量的约 79%。前期集中成交带来的市场冲击非常大，成本自然高。

这正体现了 AC 模型的核心权衡：高 λ 意味着极度规避时机风险，代价是承受更大的市场冲击。在这组参数下，AC 为了减少后期价格随机游走的方差，付出了巨额冲击成本。如果用更小的 λ（比如 1e-3），AC 会退化为接近 TWAP 的均匀分布，滑点也会接近 TWAP。

三种算法的适用场景

在决定用哪种算法之前，先想清楚你的 benchmark 是什么。

如果你的考核基准是当日 VWAP，那 VWAP 算法就是最合理的——偏差最小化就是你的目标函数，和 Almgren-Chriss 的理论最优未必一致。很多机构在绩效评估中用 VWAP 偏差来衡量执行质量，这种情况下用 TWAP 或 AC 反而会在考核上吃亏，即便它们的实际成本更低。

如果 benchmark 是决策时刻的参考价（即执行最短滑落，Implementation Shortfall），AC 框架更直接——它直接最小化 IS（Implementation Shortfall）的期望加风险加权。IS 指的是你的理论决策价和实际执行均价之间的差距，这个指标更能反映执行的真实经济成本。

如果你没有足够的数据来估计 η 和 σ，或者标的流动性足够好，TWAP 是一个体面的默认选择。简单、透明、好解释，没什么坑。

kappa 的几何意义

用图形化方式理解 AC 的最优路径可能更直观。kappa 决定了持仓下降的速度：

  • kappa × T 很小（< 0.5）：接近线性下降，几乎就是 TWAP
  • kappa × T ≈ 1.2：前期稍多，但整体还算均衡
  • kappa × T > 2：明显的凹形，前几期吃掉大半仓位

我跑了几组参数，观察到的规律是：在大多数实盘参数下，kappa × T 很难超过 1.5。除非波动率极高（比如 meme 股日内波动 5% 以上），或者市场冲击极小，否则 AC 的解和 TWAP 差异不大。这也是为什么在流动性好的市场里，TWAP 几乎是最实用的方案。

对于流动性差的小盘股或者链上资产，情况就不同了——η 可以很大，市场深度浅，稍微多成交一点就能把价格顶上去。这种场景下 AC 的价值才真正体现出来。

参数调校

AC 模型三个核心参数对行为的影响：

λ（风险厌恶）：λ → 0 时退化为 TWAP，λ → ∞ 时趋近于立刻全部执行。实际使用中，根据仓位大小和可容忍的执行误差来定，通常在 1e-6 到 1e-3 之间。

η（临时冲击）：要从历史数据里估计，通常用 LOB（限价订单簿）深度数据拟合。η 越大，AC 越倾向于分散执行。

σ（波动率）：直接用标的的历史波动率，或者用隐含波动率。注意要和期数匹配——如果每期是 30 分钟，σ 就用 30 分钟窗口的波动率。

一个测试：

// λ → 0 时退化为 TWAP
let ex = AlmgrenChrissExecutor {
    total_quantity: 1000.0,
    num_periods: 10,
    risk_aversion: 1e-12, // 接近 0
    market_impact: 0.1,
    volatility: 0.005,
    // ...
};
let traj = ex.optimal_trajectory();
// 每期量约为 100.0，与 TWAP 一致

实际使用的一些坑

参数估计是最难的部分。 AC 的数学很干净，但 η 和 σ 的估计误差会直接影响执行质量。σ 用历史窗口估会有 lag，η 更难——要从 LOB 快照里统计不同量级的冲击，数据不够干净就别想估准。有人试图用机器学习来预测盘中的临时冲击，但实际效果参差不齐，过拟合历史数据是家常便饭。

VWAP 的 profile 要实时更新。 预设 U 形 profile 只是 baseline，实际上不同标的、不同日期的 profile 差别很大。做到位的系统会维护每个标的最近 N 天的分钟级交量分布，用滑动平均来预测当天 profile。我见过一个做法是按「日历型」分类——周一开盘和周五收盘的 profile 就不一样。

执行算法是 benchmark，不是保证。 TWAP/VWAP 是和某个基准价比较的约定，不是绝对意义上的成本最小化。在实际交易中还需要考虑：是否允许被动挂单（limit order）？如何处理临时流动性不足？尾盘如何补量？这些都超出了纯执行算法的范围，需要和订单路由、风控系统配合。

子单的粒度是个隐性参数。 这篇代码里用的是连续的子单量，实际发单时要处理最小交易单位（lot size）的取整。10,000 股分 12 期，每期 833.33 股，但实际只能发整百手，累计误差会在最后一期修正。这种「尾量补偿」逻辑看似不重要，但如果系统里有多个策略同时执行，不处理好会出现实际成交量和计划量系统性偏差。
