在 Rust 里手撸 SPSC 无锁队列：原子操作、内存序与 False Sharing

⏱ 2026年3月13日
· 22 分钟阅读
#rust#concurrency#lock-free#trading#performance

我在写一个行情数据分发模块时遇到了一个老问题：市场数据线程和策略线程之间怎么高效传递 tick 数据。

std::sync::Mutex 太重，锁竞争在高频场景下会严重拉高延迟。std::sync::mpsc 是 MPSC（多生产者单消费者），在我这个场景里只有一个数据源，引入多余的原子操作开销没必要。crossbeam-channel 更通用，但我想搞清楚底层到底发生了什么，而不是一直在用黑箱。

于是就有了这篇文章。

SPSC 环形缓冲区的核心思路

SPSC（Single-Producer Single-Consumer）的前提限制反而是它的力量所在：生产者独占写端，消费者独占读端。这意味着两个游标 head（写位置）和 tail（读位置）各自只有一个线程在修改，另一个线程只是偶尔读一下。

环形缓冲区的结构很简单，用两个单调递增的整数 head 和 tail：

  • head 指向下一个写入位置
  • tail 指向下一个读取位置
  • head - tail 就是当前队列长度
  • head == tail 表示空，head - tail == capacity 表示满
  • 容量必须是 2 的幂次，这样可以用位运算代替取模：index = cursor & (capacity - 1)

整数用 usize 存，让它自然溢出回绕就行，不需要手动处理边界。

// 容量 = 4，mask = 3 (0b011)
// head = 4 时，实际槽位 = 4 & 3 = 0，完成了一次回绕

False Sharing：我踩过的最隐蔽的性能坑

第一版实现，我把 head 和 tail 就这么放在结构体里相邻的位置：

struct NaiveQueue {
    head: AtomicUsize,  // 偏移 0
    tail: AtomicUsize,  // 偏移 8
    // ...
}

基准测试结果让我困惑了一会儿：跨线程场景比单线程慢得多，但 CPU 使用率却很高，说明不是竞争导致的等待，而是 CPU 在空转干什么。

问题出在 false sharing。CPU 以 cache line（通常 64 字节）为单位操作缓存。head 和 tail 相邻，两个字段在同一条 cache line 上。生产者修改 head 时，整条 cache line 被标记为脏并失效；消费者这边缓存了 tail 的那条 cache line 被强制重新加载。反过来也一样。两个线程把彼此的缓存反复踢掉，即使它们操作的是不同字段。

修法直接：让每个字段单独占一条 cache line。

#[repr(align(64))]
struct CachePadded<T> {
    value: T,
}

pub struct SpscQueue<T> {
    head: CachePadded<AtomicUsize>, // 独占一条 cache line
    tail: CachePadded<AtomicUsize>, // 独占另一条 cache line
    capacity: usize,
    mask: usize,
    data: Box<[UnsafeCell<MaybeUninit<T>>]>,
}

#[repr(align(64))] 确保 CachePadded<T> 的起始地址按 64 字节对齐，两个字段自然落在不同 cache line 上。

内存序：不能全用 SeqCst 了事

这里是实现里最需要仔细想的地方。

内存序控制的是编译器和 CPU 对指令的重排范围。Ordering::SeqCst 最强，但代价是全局顺序屏障，在 x86 上会产生 mfence 指令，代价相当高。

SPSC 的情形其实不需要那么强的保证，用 Acquire/Release 就够了。

生产者（push）：

pub fn push(&self, value: T) -> bool {
    let head = self.head.load(Ordering::Relaxed);  // ①

    let tail = self.tail.load(Ordering::Acquire);   // ②

    if head.wrapping_sub(tail) >= self.capacity {
        return false;
    }

    let idx = head & self.mask;
    unsafe { (*self.data[idx].get()).write(value); } // ③

    self.head.store(head.wrapping_add(1), Ordering::Release); // ④
    true
}

① head 只有生产者修改，自己读自己写，Relaxed 完全够用。

② tail 是消费者写的，生产者读取时用 Acquire。这确保：当生产者看到消费者写入的 tail 值时，消费者在写 tail 之前做的所有操作（读出数据、处理数据）对生产者也可见。这里实际要的是”slot 已经被消费完，不会再被消费者访问”这个信息。

③ 写入数据，普通内存写。

④ 更新 head 用 Release，这条屏障保证：③ 的数据写入不会被重排到 head.store 之后。消费者看到新的 head 值时，③ 写入的数据也一定可见。

消费者（pop）是对称的：

pub fn pop(&self) -> Option<T> {
    let tail = self.tail.load(Ordering::Relaxed);   // ①

    let head = self.head.load(Ordering::Acquire);   // ②

    if tail == head {
        return None;
    }

    let idx = tail & self.mask;
    let value = unsafe { (*self.data[idx].get()).assume_init_read() }; // ③

    self.tail.store(tail.wrapping_add(1), Ordering::Release); // ④
    Some(value)
}

② 读 head 用 Acquire：配对生产者的 ④ Release，确保看到新 head 的同时也看到对应 slot 里的数据。

这对 Acquire/Release 就是 SPSC 正确性的核心。如果把任何一个换成 Relaxed，就可能出现消费者读到未初始化内存，或者生产者误判 slot 已释放的情况。

unsafe 在这里的合理性

UnsafeCell<MaybeUninit<T>> 是两层包装：

  • MaybeUninit<T> 告诉编译器”这块内存可能未初始化，不要对它做任何假设”，避免读取未初始化内存时的未定义行为
  • UnsafeCell<T> 关闭了 Rust 的共享引用不可变保证，允许通过 &T 修改内部数据（interior mutability）

这两个组合在 lock-free 数据结构里几乎是标配。

assume_init_read() 做的是 ptr::read()，把数据从槽位移走而不调用 Drop。这是必要的，因为槽位本身的内存由队列管理，不能在 pop 时析构。队列的 Drop 实现通过循环 pop 来确保剩余元素被正确清理。

完整代码

Cargo.toml：

[package]
name = "spsc-queue"
version = "0.1.0"
edition = "2021"
rust-version = "1.94"

src/lib.rs（核心实现，含 unsafe）：

use std::cell::UnsafeCell;
use std::mem::MaybeUninit;
use std::sync::atomic::{AtomicUsize, Ordering};

const CACHE_LINE: usize = 64;

#[repr(align(64))]
struct CachePadded<T> {
    value: T,
}

impl<T> CachePadded<T> {
    fn new(v: T) -> Self {
        CachePadded { value: v }
    }
}

impl<T> std::ops::Deref for CachePadded<T> {
    type Target = T;
    fn deref(&self) -> &T {
        &self.value
    }
}

pub struct SpscQueue<T> {
    head: CachePadded<AtomicUsize>,
    tail: CachePadded<AtomicUsize>,
    capacity: usize,
    mask: usize,
    data: Box<[UnsafeCell<MaybeUninit<T>>]>,
}

unsafe impl<T: Send> Send for SpscQueue<T> {}
unsafe impl<T: Send> Sync for SpscQueue<T> {}

impl<T> SpscQueue<T> {
    pub fn new(capacity: usize) -> Self {
        assert!(capacity.is_power_of_two(), "capacity must be a power of two, got {capacity}");
        assert!(capacity > 0);
        assert!(
            std::mem::align_of::<CachePadded<AtomicUsize>>() >= CACHE_LINE,
            "CachePadded alignment too small"
        );

        let data: Vec<UnsafeCell<MaybeUninit<T>>> = (0..capacity)
            .map(|_| UnsafeCell::new(MaybeUninit::uninit()))
            .collect();

        SpscQueue {
            head: CachePadded::new(AtomicUsize::new(0)),
            tail: CachePadded::new(AtomicUsize::new(0)),
            capacity,
            mask: capacity - 1,
            data: data.into_boxed_slice(),
        }
    }

    pub fn push(&self, value: T) -> bool {
        let head = self.head.load(Ordering::Relaxed);
        let tail = self.tail.load(Ordering::Acquire);

        if head.wrapping_sub(tail) >= self.capacity {
            return false;
        }

        let idx = head & self.mask;
        // SAFETY: 生产者独占此槽位，消费者不会同时访问
        unsafe {
            (*self.data[idx].get()).write(value);
        }

        self.head.store(head.wrapping_add(1), Ordering::Release);
        true
    }

    pub fn pop(&self) -> Option<T> {
        let tail = self.tail.load(Ordering::Relaxed);
        let head = self.head.load(Ordering::Acquire);

        if tail == head {
            return None;
        }

        let idx = tail & self.mask;
        // SAFETY: Acquire 保证看到生产者写入的完整数据
        let value = unsafe { (*self.data[idx].get()).assume_init_read() };

        self.tail.store(tail.wrapping_add(1), Ordering::Release);
        Some(value)
    }

    pub fn len(&self) -> usize {
        let head = self.head.load(Ordering::Relaxed);
        let tail = self.tail.load(Ordering::Relaxed);
        head.wrapping_sub(tail)
    }

    pub fn is_empty(&self) -> bool {
        self.len() == 0
    }

    pub fn capacity(&self) -> usize {
        self.capacity
    }
}

impl<T> Drop for SpscQueue<T> {
    fn drop(&mut self) {
        while self.pop().is_some() {}
    }
}

典型用法，生产者和消费者各自持有 Arc<SpscQueue<T>>：

use std::sync::Arc;
use std::thread;

let q = Arc::new(SpscQueue::new(4096));
let q_prod = Arc::clone(&q);

// 生产者线程
let producer = thread::spawn(move || {
    for tick in market_data_feed() {
        while !q_prod.push(tick) {
            std::hint::spin_loop(); // 队满时自旋等待
        }
    }
});

// 消费者（主线程）
loop {
    if let Some(tick) = q.pop() {
        strategy.on_tick(tick);
    } else {
        std::hint::spin_loop();
    }
}

Benchmark 结果

测试环境：Mac Studio，Apple M2 Max，32GB RAM，macOS 15.7.3，rustc 1.94.0，--release。

spsc-queue/push_pop_single_thread   time:   [9.1477 ns 9.1710 ns 9.1972 ns]
                                    thrpt:  [108.73 Melem/s 109.04 Melem/s 109.32 Melem/s]
Found 8 outliers among 100 measurements (8.00%)

spsc-queue/throughput_1k_threaded   time:   [68.104 µs 70.636 µs 73.168 µs]
                                    thrpt:  [13.667 Melem/s 14.157 Melem/s 14.683 Melem/s]

单线程 push+pop 往返约 9.2ns，折合 ~109M ops/s。这基本上是函数调用 + 两次原子操作的极限了，没什么再优化的空间。

跨线程传递 1000 个元素约 71μs，即单元素均摊 ~71ns。这里的开销主要来自 cache line 在核心间迁移（即使加了 padding，生产者更新 head 后消费者还是要拿到新值），在 ARM 上核间通信延迟大约是 40-80ns，符合预期。

跟 crossbeam-queue 比

如果你不想自己维护这段 unsafe，crossbeam-queue 的 ArrayQueue 是更好的选择，它支持 MPMC（多生产者多消费者），代价是多一些 CAS 操作。纯 SPSC 场景下我的实现会快一些，但差距不大，取决于具体场景。

这里实现的版本有一个限制：没有做真正的 Producer/Consumer 类型分离，只靠文档约定”只有一个线程调 push”。crossbeam-queue 也是这样设计的。如果想要编译时保证，需要把 SpscQueue 拆成 Producer<T> 和 Consumer<T> 两个 handle，持有各自的访问权，两个 handle 共享同一个底层 buffer（类似 Rust channel 的设计）。

在实际的行情分发系统里，我直接用 Arc wrap 然后靠架构保证单写单读，这样足够了。

关于内存序的一个提醒

这段代码在 x86/x86-64 上即使把所有 Acquire/Release 换成 Relaxed 也不会出问题，因为 x86 的 TSO 内存模型本身已经提供了 load-acquire 和 store-release 语义。但在 ARM（比如 M 系列 Mac）上是弱内存模型，Relaxed 可能导致消费者读到过时的 head 值，从而读取未初始化的槽位。

写正确的内存序，而不是”在我的机器上跑过了”。
