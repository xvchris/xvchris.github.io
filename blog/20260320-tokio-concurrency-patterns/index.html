Tokio 并发原语实战：JoinSet、CancellationToken 和 Semaphore 的生产用法

⏱ 2026年3月20日
· 17 分钟阅读
#rust#tokio#async#concurrency#tokio-conf

TokioConf 2026 下个月就要在 Portland 开了——Tokio 项目诞生十年来的第一场专属会议。趁这个机会，把我在生产项目里反复用到的几个 Tokio 并发原语整理一下。

刚开始写 async Rust 的时候，我的工具箱里只有 tokio::spawn 和 JoinHandle。后来踩了一些坑——任务泄漏、关闭时数据丢失、限流逻辑写成一坨——才发现 tokio 和 tokio-util 里其实有很多现成的原语。不复杂，但知不知道差别很大。

这篇文章把五个模式串起来，每个都有可运行的代码。

环境： Mac Studio / Apple M2 Max / 32GB，macOS 15.7.3，rustc 1.94.0，tokio 1.50.0，tokio-util 0.7.18

JoinSet：谁先回来处理谁

写行情聚合服务时，我需要同时请求多个交易所的价格 API，谁先返回就先处理谁。一开始用 Vec<JoinHandle<T>> 配合 futures::future::select_all，能跑但笨重。

JoinSet 把这件事简化了。它本质上就是一个任务集合，支持运行时动态添加任务，按完成顺序逐个取结果：

use std::time::Duration;
use tokio::task::JoinSet;
use tokio::time::sleep;

async fn fetch_price(exchange: &str, latency_ms: u64) -> (String, f64) {
    sleep(Duration::from_millis(latency_ms)).await;
    let price = match exchange {
        "binance" => 67_234.50,
        "okx" => 67_228.80,
        "bybit" => 67_241.20,
        _ => 0.0,
    };
    (exchange.to_string(), price)
}

#[tokio::main]
async fn main() {
    let mut set = JoinSet::new();

    // 运行时决定抓哪些交易所
    let exchanges = vec![("binance", 50), ("okx", 120), ("bybit", 80)];
    for (name, latency) in exchanges {
        set.spawn(fetch_price(name, latency));
    }

    // 按完成顺序处理
    while let Some(result) = set.join_next().await {
        let (exchange, price) = result.unwrap();
        println!("{exchange}: ${price:.2}");
    }
}

输出（注意顺序——binance 延迟最低先到）：

binance: $67234.50
bybit: $67241.20
okx: $67228.80

跟 Vec<JoinHandle> 比，JoinSet 的好处：不用手动管理 handle 容器，任务完成顺序自动排列，还支持在循环中继续 spawn 新任务（动态扩展）。

JoinSet 在 drop 时会取消所有未完成的任务，这个默认行为很合理——不用担心忘了清理。

CancellationToken：一个信号停掉所有 worker

我有一个行情处理管线，4 个 worker 各盯一个交易对。需要关闭时，怎么通知所有 worker 停下来？

用 channel 发关闭消息也行，但 CancellationToken 更干净：一个 token clone 给所有 worker，调一次 cancel() 全部收到：

use tokio_util::sync::CancellationToken;

async fn price_stream_worker(id: u32, token: CancellationToken) {
    let mut tick = 0u32;
    loop {
        tokio::select! {
            _ = token.cancelled() => {
                println!("worker-{id}: 收到取消信号，已处理 {tick} 条数据");
                return;
            }
            _ = tokio::time::sleep(Duration::from_millis(10)) => {
                tick += 1;
                // 处理行情数据...
            }
        }
    }
}

// 使用
let token = CancellationToken::new();
let mut set = JoinSet::new();

for id in 0..4 {
    set.spawn(price_stream_worker(id, token.clone()));
}

// 运行一段时间后关闭
tokio::time::sleep(Duration::from_millis(100)).await;
token.cancel();

CancellationToken 还支持树形结构——token.child_token() 创建子 token，取消父 token 会自动取消所有子 token，但取消子 token 不影响父。这在分层服务里很有用。

跟 channel 方案比，CancellationToken 不需要 receiver 端主动 recv，cancelled() 是一个 future，直接放进 select! 就行。

select! 取消安全：一个容易踩的坑

select! 很好用，但有个隐蔽的问题。看这段代码：

loop {
    tokio::select! {
        Some(val) = rx.recv() => {
            process(val).await;
        }
        _ = some_timer() => {
            break;
        }
    }
}

表面上没问题。但 select! 的语义是：同时 poll 所有分支，哪个先就绪走哪个，其余分支被 drop。如果某个分支的 future 在内部缓存了数据，被 drop 时数据就丢了。

mpsc::Receiver::recv() 本身是取消安全的（cancel safe）——被 drop 时不会丢消息，下次调用还能拿到。但不是所有 async 函数都有这个保证，比如 tokio::io::AsyncReadExt::read() 就不是。tokio 文档里会标注每个方法是否 cancel safe，写 select! 之前值得看一眼。

一种更稳妥的写法是把 timeout 放到外面，包住整个消费循环：

use tokio::time::timeout;

let mut count = 0u32;
let result = timeout(Duration::from_millis(200), async {
    while let Some(val) = rx.recv().await {
        count += 1;
        println!("收到: {val}");
    }
}).await;

match result {
    Ok(()) => println!("通道已关闭，共收到 {count} 条"),
    Err(_) => println!("超时退出，共收到 {count} 条"),
}

这样就没有 select! 的取消安全问题了。如果确实需要在循环里 select! 多个 channel，记得确认每个分支的 future 都是 cancel safe 的。

TaskTracker + CancellationToken：生产级 graceful shutdown

前面的 CancellationToken 只负责通知，但不保证任务真的做完了清理工作。生产环境里需要的是：通知停止 → 等待所有任务完成 → 设定超时兜底。

TaskTracker（来自 tokio-util）专门干这个。它管理一组任务的生命周期：

use tokio_util::task::TaskTracker;

let tracker = TaskTracker::new();
let token = CancellationToken::new();

for id in 0..4 {
    let t = token.clone();
    tracker.spawn(async move {
        let cleanup_ms = (id + 1) as u64 * 30;
        tokio::select! {
            _ = t.cancelled() => {
                // 收到取消后做清理
                tokio::time::sleep(Duration::from_millis(cleanup_ms)).await;
                println!("job-{id}: 清理完毕 (耗时 {cleanup_ms}ms)");
            }
            _ = tokio::time::sleep(Duration::from_secs(3600)) => {}
        }
    });
}

// 关闭 tracker，不再接受新任务
tracker.close();

// 发送取消信号
token.cancel();

// 等待所有任务完成，带超时
match timeout(Duration::from_millis(500), tracker.wait()).await {
    Ok(()) => println!("所有任务已优雅退出"),
    Err(_) => println!("警告：部分任务未在超时内完成"),
}

输出：

job-0: 清理完毕 (耗时 30ms)
job-1: 清理完毕 (耗时 60ms)
job-2: 清理完毕 (耗时 90ms)
job-3: 清理完毕 (耗时 120ms)
所有任务已优雅退出

这个三件套——CancellationToken 通知 + TaskTracker 等待 + timeout 兜底——是我现在所有长跑服务的标准关闭模式。Tokio 官方的 graceful shutdown 指南 推荐的也是这个组合。

Semaphore：控制并发不打爆上游

批量请求外部 API 时，不加限制地 spawn 几百个任务，很容易触发限流被 ban。Semaphore 是最简单的并发控制手段：

use std::sync::Arc;
use tokio::sync::Semaphore;

let semaphore = Arc::new(Semaphore::new(3)); // 最多 3 个并发
let mut set = JoinSet::new();

for id in 0..9 {
    let sem = semaphore.clone();
    set.spawn(async move {
        let _permit = sem.acquire().await.unwrap();
        // 拿到 permit 才执行请求
        let result = rate_limited_request(id).await;
        // _permit drop 时自动释放
        (id, result)
    });
}

9 个任务，每个耗时 30ms，3 并发限制下总耗时约 99ms（3 批 × 30ms），符合预期。

一些细节：

  • 别写成 let _ = sem.acquire().await——单独一个 _ 会立刻 drop permit，等于没限流。let _permit = ... 才能让 permit 活到作用域结束
  • acquire() 返回的 SemaphorePermit 实现了 Drop，不用手动释放
  • 需要”尝试获取但不等待”的语义时，用 try_acquire()
