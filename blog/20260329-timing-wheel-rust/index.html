手写 Timing Wheel：交易系统里的定时器为什么不能用 BTreeMap

⏱ 2026年3月29日
· 19 分钟阅读
#Rust#timing-wheel#trading#performance

交易系统里有个不起眼但极其关键的组件——定时器。订单超时要撤、心跳要发、限流窗口要滑动，这些东西的共同点是：数量大、插入取消频繁、延迟要可控。

我之前图省事，直接用 BTreeMap<u64, Vec<TimerId>> 来管理定时器。跑到线上以后发现个问题：当 pending 定时器到了几万量级，每次 schedule 的开销从亚微秒涨到几微秒，虽然绝对值不大，但在 tick-to-trade 链路上累积起来不可忽视。

这周花了点时间把定时器换成了 Timing Wheel 实现。记录一下过程，以及中间踩的坑。

BTreeMap 方案哪里慢

先看原来的方案：

use std::collections::BTreeMap;

type TimerId = u64;

struct BTreeTimer {
    timers: BTreeMap<u64, Vec<TimerId>>,
    next_id: TimerId,
}

impl BTreeTimer {
    fn new() -> Self {
        Self {
            timers: BTreeMap::new(),
            next_id: 0,
        }
    }

    fn schedule(&mut self, deadline_tick: u64) -> TimerId {
        let id = self.next_id;
        self.next_id += 1;
        self.timers.entry(deadline_tick).or_default().push(id);
        id
    }

    fn advance_to(&mut self, now: u64) -> Vec<TimerId> {
        let mut fired = Vec::new();
        let remaining = self.timers.split_off(&(now + 1));
        for (_tick, ids) in std::mem::replace(&mut self.timers, remaining) {
            fired.extend(ids);
        }
        fired
    }
}

这段代码功能上没问题。schedule 是 O(log N)，advance_to 用 split_off 一刀切开已到期的部分，也是 O(log N + M)（M 是触发数量）。

问题出在两个地方：

  1. BTreeMap 的节点是堆分配的，每次 insert 可能要分裂 B-tree 节点，带一次 malloc。定时器是高频插入删除的场景，分配器压力不小。
  2. cache 不友好。B-tree 的节点在内存里到处散布，pending 定时器多了以后，遍历和查找的 cache miss 率上升明显。

Timing Wheel 的核心思路

Timing Wheel 这个数据结构来自 1987 年 Varghese & Lauck 的论文（“Hashed and Hierarchical Timing Wheels”），思路简单到有点离谱：

用一个固定大小的环形数组，每个槽（slot）对应一个 tick。插入定时器时，算出它应该落在哪个 slot，O(1) 直接扔进去。每次 tick 推进，取当前 slot 里的所有条目就行，也是 O(1)。

像一个钟表——秒针走一格，看看那一格上挂了什么任务。

单层的实现极其简单：

const WHEEL_SIZE: usize = 1024;
const WHEEL_MASK: usize = WHEEL_SIZE - 1;

struct TimingWheel {
    slots: Vec<Vec<TimerId>>,
    current_tick: u64,
    next_id: TimerId,
}

impl TimingWheel {
    fn new() -> Self {
        Self {
            slots: vec![Vec::new(); WHEEL_SIZE],
            current_tick: 0,
            next_id: 0,
        }
    }

    fn schedule(&mut self, delay_ticks: u64) -> Option<TimerId> {
        if delay_ticks == 0 || delay_ticks as usize > WHEEL_SIZE {
            return None;
        }
        let id = self.next_id;
        self.next_id += 1;
        let slot_idx = (self.current_tick + delay_ticks) as usize & WHEEL_MASK;
        self.slots[slot_idx].push(id);
        Some(id)
    }

    fn tick(&mut self) -> Vec<TimerId> {
        self.current_tick += 1;
        let slot_idx = self.current_tick as usize & WHEEL_MASK;
        std::mem::take(&mut self.slots[slot_idx])
    }
}

WHEEL_SIZE 选 1024（2 的幂），slot 索引直接 & WHEEL_MASK，位运算搞定，没有取模的开销。

但单层有个明显的限制：时间跨度被 WHEEL_SIZE 卡死了。1024 个 slot，如果每个 tick 是 1ms，那最远只能调度 1 秒后的事件。交易系统里订单超时一般 30 秒到几分钟，单层根本不够用。

两级层次时间轮

解决方案是加层级。就像钟表有秒针和分针一样——level 0 管细粒度（每 slot = 1 tick），level 1 管粗粒度（每 slot = 1024 ticks）。两层叠起来能覆盖 1024 × 1024 = 1,048,576 个 tick。

const LEVEL_BITS: u32 = 10;
const LEVEL_SIZE: usize = 1 << LEVEL_BITS; // 1024
const LEVEL_MASK: usize = LEVEL_SIZE - 1;

#[derive(Clone)]
struct WheelEntry {
    id: TimerId,
    deadline: u64,
}

struct HierarchicalWheel {
    level0: Vec<Vec<WheelEntry>>,
    level1: Vec<Vec<WheelEntry>>,
    current_tick: u64,
    next_id: TimerId,
}

插入时根据 delay 大小决定放哪层：

impl HierarchicalWheel {
    fn new() -> Self {
        Self {
            level0: vec![Vec::new(); LEVEL_SIZE],
            level1: vec![Vec::new(); LEVEL_SIZE],
            current_tick: 0,
            next_id: 0,
        }
    }

    fn schedule(&mut self, delay_ticks: u64) -> Option<TimerId> {
        if delay_ticks == 0 {
            return None;
        }
        let max_delay = (LEVEL_SIZE * LEVEL_SIZE) as u64;
        if delay_ticks > max_delay {
            return None;
        }
        let id = self.next_id;
        self.next_id += 1;
        let deadline = self.current_tick + delay_ticks;
        let entry = WheelEntry { id, deadline };
        self.insert_entry(entry);
        Some(id)
    }

    fn insert_entry(&mut self, entry: WheelEntry) {
        let delta = entry.deadline - self.current_tick;
        if delta <= LEVEL_SIZE as u64 {
            let slot = entry.deadline as usize & LEVEL_MASK;
            self.level0[slot].push(entry);
        } else {
            let slot = (entry.deadline >> LEVEL_BITS) as usize & LEVEL_MASK;
            self.level1[slot].push(entry);
        }
    }
}

关键在 tick 推进时的 cascade（级联下沉）：

    fn tick(&mut self) -> Vec<TimerId> {
        self.current_tick += 1;
        let slot0 = self.current_tick as usize & LEVEL_MASK;

        // level 0 转了一圈，从 level 1 拉下来
        if slot0 == 0 {
            let slot1 = (self.current_tick >> LEVEL_BITS) as usize & LEVEL_MASK;
            let entries = std::mem::take(&mut self.level1[slot1]);
            for entry in entries {
                self.insert_entry(entry);
            }
        }

        let fired_entries = std::mem::take(&mut self.level0[slot0]);
        fired_entries.into_iter().map(|e| e.id).collect()
    }

当 level 0 走完一整圈（slot0 回到 0），从 level 1 对应的 slot 里取出所有 entry，重新 insert——因为它们现在离 deadline 已经足够近，会被分配到 level 0 的精确位置上。

cascade 的频率是每 1024 个 tick 发生一次。如果 tick 间隔是 1ms，那大约每秒 cascade 一次，完全可接受。

在 Mac Studio (Apple M2 Max, 32 GB) 上跑 benchmark，Rust 1.94.0，macOS 26.3.1：

插入 10K 定时器

实现耗时
BTreeMap535 µs
Hierarchical Wheel42.6 µs

快 12.6 倍。

推进 1K tick（每 tick 触发 1 个定时器）

实现耗时
BTreeMap186.7 µs
Hierarchical Wheel11.6 µs

快 16.1 倍。

BTreeMap 在 tick 推进上用的是 split_off，理论上对有序数据应该快。实际上 split_off 每次都要找分裂点、重新平衡树，这些操作的 cache miss 代价在高频调用时很突出。

Timing Wheel 的 tick 就是一次数组下标访问 + mem::take，几乎没有间接寻址。

cascade 的开销问题

有一点要注意：cascade 发生时，一个 level 1 slot 里可能积攒了不少 entry。如果某个时间段内集中调度了大量定时器，cascade 那一个 tick 会出现延迟尖峰。

在交易系统里的实际情况是——订单超时通常散布在几秒到几十秒的范围内，不太会集中到同一个 level 1 slot。而且 cascade 只是把 entry 从一个 Vec 移到另一个 Vec，没有堆分配（Vec 的容量还在），开销可控。

如果业务场景确实有大量定时器集中在同一时间窗口（比如整秒或整分触发），可以考虑加到三层，或者在 cascade 时分摊处理。Linux 内核的 timer wheel 就用了类似的思路，分了 4 层（tvec_base_s）。

取消定时器

上面的实现有个缺陷——没有 cancel。实际用的时候，订单被主动撤单了，对应的超时定时器需要取消。

一个简单做法是给每个 entry 加一个 cancelled: bool 标记，cancel 时设 true，tick 触发时跳过。这会有一定的内存浪费（cancelled 的 entry 还占着 slot），但避免了在 Vec 里做 O(N) 查找和删除。

更精细的做法是用一个 HashMap<TimerId, (level, slot)> 做反查索引，cancel 时直接定位到 entry。我目前用的是标记法，够用了，等有性能压力再换。
