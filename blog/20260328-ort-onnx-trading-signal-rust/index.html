用 ort 在 Rust 里跑 ONNX 模型：交易信号推理的 3 微秒之路

⏱ 2026年3月28日
· 27 分钟阅读
#rust#onnx#ort#ml-inference#trading

做交易系统的时候，ML 模型推理是一个绕不开的问题。Python 训练模型方便，但推理延迟太高——一个 sklearn 模型用 predict_proba 跑一次都要几十微秒，加上 GIL 和 GC 的抖动，放在实时系统里很不舒服。

我最近在尝试一条路径：Python 训练，ONNX 导出，Rust 推理。用的是 ort 这个 crate，ONNX Runtime 的 Rust binding。跑下来单次推理 3.46 µs，batch 模式下每个 sample 摊到 0.66 µs。记录一下过程和踩的坑。

为什么是 ONNX

ONNX 本身就是为了解决”训练和推理用不同语言”这个问题设计的。模型训练用 Python（sklearn、PyTorch、XGBoost 都行），导出成 .onnx 文件，推理端用 C++/Rust/Go 加载。对交易系统来说，这意味着：

  • 策略研究员继续用 Python 和 Jupyter，不用学 Rust
  • 推理端是纯 Rust，延迟可控，没有 Python runtime 的开销
  • 模型更新只需要替换 .onnx 文件，不用重新编译

模型准备

先在 Python 端训练一个简单的信号分类器。3 个类别：short / hold / long。特征用了 4 个：价格变化率、成交量比率、RSI 归一化值、spread（基点）。

import numpy as np
from sklearn.neural_network import MLPClassifier

np.random.seed(42)
n_samples = 5000
features = np.random.randn(n_samples, 4).astype(np.float32)

# 标签规则：RSI 低 + spread 大 → long，RSI 高 + spread 小 → short，其余 hold
labels = np.ones(n_samples, dtype=np.int64)
labels[(features[:, 2] < -0.5) & (features[:, 3] > 0.3)] = 2  # long
labels[(features[:, 2] > 0.5) & (features[:, 3] < -0.3)] = 0  # short

clf = MLPClassifier(hidden_layer_sizes=(32, 16), max_iter=500, random_state=42)
clf.fit(features, labels)
# Training accuracy: 0.9322

这里用合成数据只是为了演示流程。实际场景中特征会复杂得多，但 ONNX 导出和推理的部分是一样的。

导出 ONNX 的坑

sklearn 有个官方工具 skl2onnx 可以直接把模型转成 ONNX。但这里有个大坑：sklearn 导出的分类器，概率输出是 Sequence(Map(int64, float)) 类型。

这个格式在 Python 的 onnxruntime 里没问题，但 ort crate 2.0.0-rc.12 对 Sequence(Map) 的支持还没实现完——调用 try_extract_sequence 之后再 try_extract_map 会直接 panic，报 not yet implemented。

翻了一下 ort 的源码，src/value/type.rs:186 那里确实是个 todo!()。

绕过这个问题的办法：不用 skl2onnx，手动构建 ONNX 图。把 sklearn 训练好的权重提取出来，用 onnx.helper 搭一个等价的计算图，输出直接是 float32 tensor。

import onnx
from onnx import helper, TensorProto, numpy_helper

# 从 sklearn 模型提取权重
W1 = clf.coefs_[0].astype(np.float32)   # (4, 32)
b1 = clf.intercepts_[0].astype(np.float32)
W2 = clf.coefs_[1].astype(np.float32)   # (32, 16)
b2 = clf.intercepts_[1].astype(np.float32)
W3 = clf.coefs_[2].astype(np.float32)   # (16, 3)
b3 = clf.intercepts_[2].astype(np.float32)

X = helper.make_tensor_value_info('features', TensorProto.FLOAT, [None, 4])
Y = helper.make_tensor_value_info('probabilities', TensorProto.FLOAT, [None, 3])

initializers = [
    numpy_helper.from_array(W1, 'W1'), numpy_helper.from_array(b1, 'b1'),
    numpy_helper.from_array(W2, 'W2'), numpy_helper.from_array(b2, 'b2'),
    numpy_helper.from_array(W3, 'W3'), numpy_helper.from_array(b3, 'b3'),
]

nodes = [
    helper.make_node('MatMul', ['features', 'W1'], ['h1_mm']),
    helper.make_node('Add', ['h1_mm', 'b1'], ['h1']),
    helper.make_node('Relu', ['h1'], ['h1_act']),
    helper.make_node('MatMul', ['h1_act', 'W2'], ['h2_mm']),
    helper.make_node('Add', ['h2_mm', 'b2'], ['h2']),
    helper.make_node('Relu', ['h2'], ['h2_act']),
    helper.make_node('MatMul', ['h2_act', 'W3'], ['logits_mm']),
    helper.make_node('Add', ['logits_mm', 'b3'], ['logits']),
    helper.make_node('Softmax', ['logits'], ['probabilities'], axis=1),
]

graph = helper.make_graph(nodes, 'signal_predictor', [X], [Y], initializers)
model = helper.make_model(graph, opset_imports=[helper.make_opsetid('', 13)])
onnx.save(model, 'signal_model.onnx')

这样导出的模型只有 3KB 左右，输出就是干净的 [batch_size, 3] float32 tensor。

Rust 端：ort crate

ort 是 ONNX Runtime 的 Rust wrapper，目前最新是 2.0.0-rc.12。它会在首次编译时自动下载预编译的 ONNX Runtime 动态库，不需要自己装。

[dependencies]
ort = "2.0.0-rc.12"
ndarray = "0.17"

一个要注意的点：ndarray 版本必须和 ort 内部依赖的版本对齐。ort 2.0.0-rc.12 依赖 ndarray 0.17，如果你在 Cargo.toml 里写 ndarray = "0.16"，编译能过，但 Tensor::from_array 会报 trait bound 不满足——因为 ort 的 OwnedTensorArrayData 是给 ndarray 0.17 的 Array 实现的。错误信息里会提示 “there are multiple different versions of crate ndarray in the dependency graph”，看到这个就知道版本对不上了。

推理代码

use ndarray::Array2;
use ort::{session::Session, value::Tensor};

const NUM_FEATURES: usize = 4;
const NUM_CLASSES: usize = 3;
const CLASS_NAMES: [&str; NUM_CLASSES] = ["short", "hold", "long"];

fn load_model(path: &str) -> ort::Result<Session> {
    Session::builder()?.commit_from_file(path)
}

fn predict(
    session: &mut Session,
    features: &[f32; NUM_FEATURES],
) -> ort::Result<(usize, &'static str, f32)> {
    let input = Array2::from_shape_vec(
        (1, NUM_FEATURES), features.to_vec()
    ).unwrap();
    let input_tensor = Tensor::from_array(input)?;
    let outputs = session.run(ort::inputs![input_tensor])?;

    // output shape: [1, 3]，try_extract_tensor 返回 (Shape, &[f32])
    let (_, probs_data) = outputs[0].try_extract_tensor::<f32>()?;
    let probs = &probs_data[..NUM_CLASSES];
    let (class_idx, &confidence) = probs
        .iter()
        .enumerate()
        .max_by(|(_, a), (_, b)| a.partial_cmp(b).unwrap())
        .unwrap();

    Ok((class_idx, CLASS_NAMES[class_idx], confidence))
}

几个 ort 2.x 的 API 注意点：

  1. Session::run 需要 &mut self。这意味着如果你要在多线程里共享 Session，得用 Mutex<Session> 或者每个线程各自持有一个。对交易系统来说，每个策略线程加载自己的 Session 更合理，避免锁竞争。

  2. try_extract_tensor::<T>() 返回的是 (Shape, &[T]) 元组，不是 ndarray 的 ArrayView。Shape 里有维度信息，数据是扁平的切片。

  3. ort::inputs! 宏用来构造输入，按模型定义的顺序传入。不需要指定名字，按位置匹配。

Batch 推理

实际用的时候，经常需要对多个 symbol 同时做推理。batch 模式下 ONNX Runtime 的内部并行会发挥作用：

fn predict_batch(
    session: &mut Session,
    batch: &[[f32; NUM_FEATURES]],
) -> ort::Result<Vec<(usize, &'static str, f32)>> {
    let n = batch.len();
    let flat: Vec<f32> = batch
        .iter()
        .flat_map(|f| f.iter().copied())
        .collect();
    let input = Array2::from_shape_vec((n, NUM_FEATURES), flat).unwrap();
    let input_tensor = Tensor::from_array(input)?;
    let outputs = session.run(ort::inputs![input_tensor])?;

    let (_, probs_data) = outputs[0].try_extract_tensor::<f32>()?;
    let mut results = Vec::with_capacity(n);
    for i in 0..n {
        let probs = &probs_data[i * NUM_CLASSES..(i + 1) * NUM_CLASSES];
        let (class_idx, &confidence) = probs
            .iter()
            .enumerate()
            .max_by(|(_, a), (_, b)| a.partial_cmp(b).unwrap())
            .unwrap();
        results.push((class_idx, CLASS_NAMES[class_idx], confidence));
    }
    Ok(results)
}

输出解析那里用切片索引就行，probs_data 是按行展开的，每 NUM_CLASSES 个元素对应一个 sample。

性能数据

测试环境：

  • Mac Studio, Apple M2 Max, 32 GB
  • macOS 26.3.1
  • rustc 1.94.0 (4a4ef493e 2026-03-02)
  • ort 2.0.0-rc.12 (ONNX Runtime 1.24)
  • 模型：3 层 MLP，4→32→16→3，约 3KB
Single-inference latency (10000 iters):
  Total: 34.61ms
  Per call: 3.46 µs

Batch-4 latency (10000 iters):
  Total: 26.21ms
  Per sample (amortized): 0.66 µs

3.46 µs 的单次推理延迟，对于行情 tick 驱动的策略来说足够用了。大部分加密货币交易所的 WebSocket 推送间隔在毫秒级，推理延迟占比不到 1%。

batch 模式下每个 sample 摊到 0.66 µs，差不多是 5 倍的加速。这主要是省掉了重复的 Session 调用开销和内存分配。如果你同时跑 20 个交易对，一次 batch 推理比循环调 20 次 predict 快很多。

实际集成的考虑

在真实交易系统里用，还有几个点要处理：

模型热更新：最简单的做法是 watch .onnx 文件的修改时间，变了就重新 load_model。ort 的 Session 可以 drop 再重建，不需要重启进程。

特征工程：上面例子里的 extract_features 是硬编码的。实际中我会配合之前写的在线统计模块（Welford 算法那篇）算 RSI 和 volume ratio，避免用 Vec 存全量历史。

置信度阈值：不是每个推理结果都该执行。我一般设一个 confidence 阈值（比如 0.7），低于这个的输出 hold。模型给出 short 0.35 / hold 0.33 / long 0.32 这种”三个都差不多”的结果，说明它也不确定，不下注比较安全。

模型监控：跑一段时间之后要看推理结果的分布有没有漂移。如果某天突然 90% 都是 long，要么市场真的单边了，要么特征管道出了 bug。

完整代码

Cargo.toml:

[package]
name = "ort_inference"
version = "0.1.0"
edition = "2021"

[dependencies]
ort = "2.0.0-rc.12"
ndarray = "0.17"

main.rs:

use std::time::Instant;

use ndarray::Array2;
use ort::{session::Session, value::Tensor};

const NUM_FEATURES: usize = 4;
const NUM_CLASSES: usize = 3;
const CLASS_NAMES: [&str; NUM_CLASSES] = ["short", "hold", "long"];

fn load_model(path: &str) -> ort::Result<Session> {
    Session::builder()?.commit_from_file(path)
}

fn predict(
    session: &mut Session,
    features: &[f32; NUM_FEATURES],
) -> ort::Result<(usize, &'static str, f32)> {
    let input = Array2::from_shape_vec(
        (1, NUM_FEATURES), features.to_vec()
    ).unwrap();
    let input_tensor = Tensor::from_array(input)?;
    let outputs = session.run(ort::inputs![input_tensor])?;

    let (_, probs_data) = outputs[0].try_extract_tensor::<f32>()?;
    let probs = &probs_data[..NUM_CLASSES];
    let (class_idx, &confidence) = probs
        .iter()
        .enumerate()
        .max_by(|(_, a), (_, b)| a.partial_cmp(b).unwrap())
        .unwrap();

    Ok((class_idx, CLASS_NAMES[class_idx], confidence))
}

fn predict_batch(
    session: &mut Session,
    batch: &[[f32; NUM_FEATURES]],
) -> ort::Result<Vec<(usize, &'static str, f32)>> {
    let n = batch.len();
    let flat: Vec<f32> = batch
        .iter()
        .flat_map(|f| f.iter().copied())
        .collect();
    let input = Array2::from_shape_vec((n, NUM_FEATURES), flat).unwrap();
    let input_tensor = Tensor::from_array(input)?;
    let outputs = session.run(ort::inputs![input_tensor])?;

    let (_, probs_data) = outputs[0].try_extract_tensor::<f32>()?;
    let mut results = Vec::with_capacity(n);
    for i in 0..n {
        let probs = &probs_data[i * NUM_CLASSES..(i + 1) * NUM_CLASSES];
        let (class_idx, &confidence) = probs
            .iter()
            .enumerate()
            .max_by(|(_, a), (_, b)| a.partial_cmp(b).unwrap())
            .unwrap();
        results.push((class_idx, CLASS_NAMES[class_idx], confidence));
    }
    Ok(results)
}

fn main() -> ort::Result<()> {
    let mut session = load_model("signal_model.onnx")?;

    let features = [0.1f32, 0.5, -0.8, 0.6];
    let (idx, name, conf) = predict(&mut session, &features)?;
    println!("Single: class={idx} ({name}), confidence={conf:.4}");

    let batch = vec![
        [0.1f32, 0.5, -0.8, 0.6],
        [-0.3, 1.2, 0.5, -0.1],
        [0.0, 0.8, -1.2, 0.9],
        [0.5, -0.5, 0.8, -0.6],
    ];
    let results = predict_batch(&mut session, &batch)?;
    for (i, (idx, name, conf)) in results.iter().enumerate() {
        println!("  sample {i}: class={idx} ({name}), confidence={conf:.4}");
    }

    // benchmark
    let n_iters = 10_000;
    let start = Instant::now();
    for _ in 0..n_iters {
        let _ = predict(&mut session, &features)?;
    }
    let elapsed = start.elapsed();
    println!(
        "Per inference: {:.2} µs",
        elapsed.as_micros() as f64 / n_iters as f64
    );

    Ok(())
}
