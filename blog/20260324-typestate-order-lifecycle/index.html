用 Typestate Pattern 把订单状态机焊死在编译期

⏱ 2026年3月24日
· 29 分钟阅读
#Rust#类型系统#状态机#交易系统

去年写交易系统的时候，我被一个 bug 折腾了两天：一个已经全部成交的订单，因为回调时序问题，又被撤单逻辑处理了一遍。账务系统认为这笔钱释放了，策略层又重新下了单。等我发现的时候，已经多出了一堆幽灵仓位。

根因很简单——订单的状态是一个 enum，转换逻辑散落在各个回调里，靠 if order.status == ... 来守护。漏了一个判断就炸了。

这篇文章讲的是我后来用 typestate pattern 重构订单状态机的过程。核心思路：把状态编码到类型参数里，让编译器来保证不合法的转换写不出来。

问题：enum 状态机哪里不好

先看一个典型的 enum 实现：

enum OrderStatus {
    Draft,
    Submitted,
    Accepted,
    PartialFill,
    Filled,
    Cancelled,
}

struct Order {
    status: OrderStatus,
    // ...
}

impl Order {
    fn cancel(&mut self) -> Result<(), &'static str> {
        match self.status {
            OrderStatus::Accepted | OrderStatus::PartialFill => {
                self.status = OrderStatus::Cancelled;
                Ok(())
            }
            _ => Err("cannot cancel in current state"),
        }
    }
}

这能跑，但有几个问题：

  1. 状态检查靠人。每个操作函数都要 match 当前状态，漏一个就是运行时 panic 或者静默错误。
  2. 编译器帮不上忙。你写 order.cancel() 的时候，编译器不知道这个 order 处于什么状态，它只能说”好的，你调了 cancel，返回 Result”。
  3. 测试覆盖不住所有路径。状态多了之后，排列组合爆炸，总有你没想到的边界情况。

我那个 bug 就是第一类——PartialFill 状态下收到全成交回报，先转 Filled，然后一个迟到的撤单响应进来，代码没有 re-check 状态就执行了撤单逻辑。

Typestate：把状态塞进泛型参数

思路其实不复杂：每个状态是一个独立的类型，Order<Draft> 和 Order<Filled> 是两个不同的类型。方法只定义在对应状态的 impl 块里，其他状态根本没有这个方法。

先定义状态：

use std::marker::PhantomData;

struct Draft;
struct Submitted;
struct Accepted;
struct PartialFill;
struct Filled;
struct Cancelled;

六个空 struct，不占内存，纯粹用来做类型标记。

订单主体：

#[derive(Debug, Clone)]
struct OrderInfo {
    symbol: String,
    side: Side,
    price: f64,
    quantity: f64,
    filled_qty: f64,
    order_id: Option<String>,
}

#[derive(Debug, Clone, Copy, PartialEq)]
enum Side {
    Buy,
    Sell,
}

struct Order<S> {
    info: OrderInfo,
    _state: PhantomData<S>,
}

PhantomData<S> 是关键。它告诉编译器”这个结构体在类型层面跟 S 有关系”，但运行时不占任何空间。Order<Draft> 和 Order<Filled> 在内存布局上完全一样，区别只存在于类型系统里。

状态转换：消费 self，返回新类型

状态转换的核心：方法 consume 旧的 Order<A>，返回新的 Order<B>。

先写一个通用的 transition 方法：

impl<S> Order<S> {
    fn transition<T>(self) -> Order<T> {
        Order {
            info: self.info,
            _state: PhantomData,
        }
    }

    fn transition_with<T>(self, f: impl FnOnce(&mut OrderInfo)) -> Order<T> {
        let mut info = self.info;
        f(&mut info);
        Order {
            info,
            _state: PhantomData,
        }
    }
}

transition 只改类型不改数据，transition_with 允许在转换时修改字段（比如记录 order_id 或更新已成交数量）。

因为 self 是 move 语义，调用 transition 之后原来的 Order<A> 就不存在了。不可能出现”旧状态还被引用”的问题。

按状态定义方法

这是 typestate 的核心。submit 只在 Order<Draft> 上存在：

impl Order<Draft> {
    fn new(symbol: &str, side: Side, price: f64, quantity: f64) -> Self {
        Order {
            info: OrderInfo {
                symbol: symbol.to_string(),
                side,
                price,
                quantity,
                filled_qty: 0.0,
                order_id: None,
            },
            _state: PhantomData,
        }
    }

    fn submit(self) -> Order<Submitted> {
        self.transition()
    }
}

accept 和 reject 只在 Order<Submitted> 上：

impl Order<Submitted> {
    fn accept(self, order_id: &str) -> Order<Accepted> {
        self.transition_with(|info| {
            info.order_id = Some(order_id.to_string());
        })
    }

    fn reject(self, reason: &str) -> Order<Cancelled> {
        self.transition()
    }
}

fill 和 cancel 在 Order<Accepted> 上：

impl Order<Accepted> {
    fn fill(self, qty: f64) -> FillResult {
        let new_filled = self.info.filled_qty + qty;
        if new_filled >= self.info.quantity {
            let order = self.transition_with(|info| {
                info.filled_qty = info.quantity;
            });
            FillResult::Full(order)
        } else {
            let order = self.transition_with(|info| {
                info.filled_qty = new_filled;
            });
            FillResult::Partial(order)
        }
    }

    fn cancel(self) -> Order<Cancelled> {
        self.transition()
    }
}

fill 的返回值是个 enum——因为成交之后可能是全成交也可能是部分成交，编译器没法在类型层面静态确定：

enum FillResult {
    Partial(Order<PartialFill>),
    Full(Order<Filled>),
}

调用方必须 match 处理两种情况，忘不了。

PartialFill 也能继续 fill 或者 cancel：

impl Order<PartialFill> {
    fn fill(self, qty: f64) -> FillResult {
        let new_filled = self.info.filled_qty + qty;
        if new_filled >= self.info.quantity {
            let order = self.transition_with(|info| {
                info.filled_qty = info.quantity;
            });
            FillResult::Full(order)
        } else {
            let order = self.transition_with(|info| {
                info.filled_qty = new_filled;
            });
            FillResult::Partial(order)
        }
    }

    fn cancel(self) -> Order<Cancelled> {
        self.transition()
    }
}

终态 Filled 和 Cancelled 只有读方法，不能再做任何操作：

impl Order<Filled> {
    fn summary(&self) -> String {
        format!("{} {:?} {} 全部成交",
            self.info.symbol, self.info.side, self.info.quantity)
    }
}

impl Order<Cancelled> {
    fn summary(&self) -> String {
        format!("{} 已撤单 (已成交 {})",
            self.info.symbol, self.info.filled_qty)
    }
}

编译器替你守护

现在试试写非法操作：

let draft = Order::<Draft>::new("BTC-USDT", Side::Buy, 65000.0, 1.0);
draft.cancel(); // 编译错误：Order<Draft> 没有 cancel 方法
error[E0599]: no method named `cancel` found for struct `Order<Draft>`

再来一个：

let filled: Order<Filled> = /* ... */;
filled.fill(1.0); // 编译错误：Order<Filled> 没有 fill 方法

我那个 bug 的场景——对已经 Filled 的订单执行撤单——直接写不出来。不是运行时报错，是代码根本过不了编译。

实际用起来的样子

正常全成交：

let order = Order::<Draft>::new("BTC-USDT", Side::Buy, 65000.0, 1.0);
let order = order.submit();
let order = order.accept("ORD-001");

match order.fill(1.0) {
    FillResult::Full(filled) => {
        println!("{}", filled.summary());
        // BTC-USDT Buy 1 全部成交
    }
    FillResult::Partial(partial) => {
        // 继续等后续成交...
    }
}

部分成交后撤单：

let order = Order::<Draft>::new("ETH-USDT", Side::Sell, 3500.0, 10.0);
let order = order.submit();
let order = order.accept("ORD-002");

let order = match order.fill(3.0) {
    FillResult::Partial(p) => p,
    FillResult::Full(f) => { /* 处理全成交 */ return; }
};

let order = match order.fill(2.0) {
    FillResult::Partial(p) => p,
    FillResult::Full(f) => { /* 处理全成交 */ return; }
};

// 不等了，撤掉
let cancelled = order.cancel();
// ETH-USDT 已撤单 (已成交 5)

注意每一步的 order 都是 let 重新绑定的新变量，旧的被 move 掉了。Rust 的所有权系统刚好配合 typestate 用——你不可能偷偷保留一个旧状态的引用。

这个模式的边界在哪

用了几个月，我觉得 typestate 适合这些场景：

  • 状态数量有限且明确（6-8 个以内比较舒服）
  • 状态转换规则固定，不会频繁变化
  • 非法转换的后果严重（资金相关、安全相关）

不太适合的：

状态需要运行时动态判断。 比如你有一个 Vec<Order<?>>，里面混着各种状态的订单——这写不了。泛型参数在编译期确定，运行时你没法搞一个”任意状态的订单”集合。解决方案是搞一个包装 enum：

enum AnyOrder {
    Draft(Order<Draft>),
    Submitted(Order<Submitted>),
    Accepted(Order<Accepted>),
    // ...
}

但这又回到了 match 的路子，只是内部每个分支的操作仍然有类型保证。

状态转换有分叉且分叉特别多。 FillResult 这种两分叉还好，如果一个操作有五六种可能的结果类型，match 层层嵌套会很烦。

跨线程/跨异步边界。 如果订单对象要在多个 async task 之间传递，状态转换的 move 语义会跟 Arc<Mutex<>> 之类的共享机制冲突。这时候可能 enum + 内部检查反而更实用。

零成本抽象

最后说一下性能。PhantomData<S> 是零大小类型，Order<Draft> 和 Order<Filled> 的内存布局完全相同。状态转换就是一个 move，没有额外的分配、拷贝或运行时检查。

这就是 Rust 类型系统做得好的地方——安全检查全在编译期完成，运行时开销是零。enum 方案每次操作都要 match 一下当前状态，typestate 方案连这个 match 都省了，因为类型本身就是状态。

完整代码

完整的可运行代码放在这里，cargo new typestate_order 之后替换 main.rs 就能跑：

use std::marker::PhantomData;

struct Draft;
struct Submitted;
struct Accepted;
struct PartialFill;
struct Filled;
struct Cancelled;

#[derive(Debug, Clone)]
struct OrderInfo {
    symbol: String,
    side: Side,
    price: f64,
    quantity: f64,
    filled_qty: f64,
    order_id: Option<String>,
}

#[derive(Debug, Clone, Copy, PartialEq)]
enum Side {
    Buy,
    Sell,
}

struct Order<S> {
    info: OrderInfo,
    _state: PhantomData<S>,
}

impl<S> Order<S> {
    fn filled_qty(&self) -> f64 { self.info.filled_qty }
    fn quantity(&self) -> f64 { self.info.quantity }

    fn transition<T>(self) -> Order<T> {
        Order { info: self.info, _state: PhantomData }
    }

    fn transition_with<T>(self, f: impl FnOnce(&mut OrderInfo)) -> Order<T> {
        let mut info = self.info;
        f(&mut info);
        Order { info, _state: PhantomData }
    }
}

impl Order<Draft> {
    fn new(symbol: &str, side: Side, price: f64, quantity: f64) -> Self {
        Order {
            info: OrderInfo {
                symbol: symbol.to_string(), side, price, quantity,
                filled_qty: 0.0, order_id: None,
            },
            _state: PhantomData,
        }
    }

    fn submit(self) -> Order<Submitted> { self.transition() }
}

impl Order<Submitted> {
    fn accept(self, order_id: &str) -> Order<Accepted> {
        self.transition_with(|info| { info.order_id = Some(order_id.to_string()); })
    }

    fn reject(self, _reason: &str) -> Order<Cancelled> { self.transition() }
}

enum FillResult {
    Partial(Order<PartialFill>),
    Full(Order<Filled>),
}

fn do_fill<S>(order: Order<S>, qty: f64) -> FillResult {
    let new_filled = order.info.filled_qty + qty;
    if new_filled >= order.info.quantity {
        FillResult::Full(order.transition_with(|i| i.filled_qty = i.quantity))
    } else {
        FillResult::Partial(order.transition_with(|i| i.filled_qty = new_filled))
    }
}

impl Order<Accepted> {
    fn fill(self, qty: f64) -> FillResult { do_fill(self, qty) }
    fn cancel(self) -> Order<Cancelled> { self.transition() }
}

impl Order<PartialFill> {
    fn fill(self, qty: f64) -> FillResult { do_fill(self, qty) }
    fn cancel(self) -> Order<Cancelled> { self.transition() }
}

impl Order<Filled> {
    fn summary(&self) -> String {
        format!("{} {:?} {} 全部成交",
            self.info.symbol, self.info.side, self.info.quantity)
    }
}

impl Order<Cancelled> {
    fn summary(&self) -> String {
        format!("{} 已撤单 (已成交 {})", self.info.symbol, self.info.filled_qty)
    }
}

fn main() {
    // 正常全成交
    let order = Order::<Draft>::new("BTC-USDT", Side::Buy, 65000.0, 1.0);
    let order = order.submit();
    let order = order.accept("ORD-001");
    match order.fill(1.0) {
        FillResult::Full(f) => println!("{}", f.summary()),
        FillResult::Partial(p) => println!("剩余 {}", p.quantity() - p.filled_qty()),
    }

    // 部分成交后撤单
    let order = Order::<Draft>::new("ETH-USDT", Side::Sell, 3500.0, 10.0);
    let order = order.submit();
    let order = order.accept("ORD-002");
    let order = match order.fill(3.0) {
        FillResult::Partial(p) => p,
        FillResult::Full(f) => { println!("{}", f.summary()); return; }
    };
    let order = match order.fill(2.0) {
        FillResult::Partial(p) => p,
        FillResult::Full(f) => { println!("{}", f.summary()); return; }
    };
    let cancelled = order.cancel();
    println!("{}", cancelled.summary());
}
BTC-USDT Buy 1 全部成交
ETH-USDT 已撤单 (已成交 5)
