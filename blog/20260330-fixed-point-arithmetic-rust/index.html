交易系统里为什么不能用 f64 算钱：Rust 定点数实现

⏱ 2026年3月30日
· 17 分钟阅读
#rust#fixed-point#trading#precision

做加密货币交易系统的时候，有个问题迟早要面对：钱怎么存？

我最初图省事用 f64，跑了几天回测发现一个诡异的 bug——累计手续费和交易所账单对不上。差距不大，小数点后第 7 位开始偏，但 100 万笔交易下来就能差出几个 satoshi。排查了半天，原因很简单：IEEE 754 浮点数没法精确表示 0.1。

这篇记录一下我后来用的方案：i64 定点数，精度 10^-8，刚好覆盖 BTC 的 8 位小数。

f64 到底差在哪

先看经典的：

let a: f64 = 0.1;
let b: f64 = 0.2;
println!("{:.20}", a + b);
// 0.30000000000000004441

0.1 在二进制里是无限循环小数，存进 64 位浮点必然截断。单次运算误差在 10^-16 量级，看着可以忽略。但交易系统的特点是大量小额运算反复累加——手续费、资金费率、滑点扣减，每笔都是亚 satoshi 级别的数。

实测：把 37 satoshi（0.00000037 BTC）的手续费累加 100 万次：

f64 结果:   0.37000000（看起来对？）
精确值:     0.37000000
f64 误差:   8.55e-12

100 万笔才差 10^-12，好像还行？问题是这个误差不均匀。某些金额组合会放大误差，而且浮点比较天然不靠谱——你永远不知道 a + b == c 会不会因为精度翻车。

在对账的时候，交易所返回的是字符串类型的精确小数。你用 f64 算出来的和人家对不上，哪怕差一个最小单位，那也是差。

定点数方案

思路很直接：用整数存，约定一个缩放因子。BTC 最小单位是 satoshi（10^-8），那就用 10^8 做缩放因子，所有金额都乘以 10^8 存成 i64。

#[derive(Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash)]
struct FixedPoint {
    raw: i64,
}

const SCALE: i64 = 100_000_000; // 10^8
const SCALE_F64: f64 = 100_000_000.0;

为什么选 i64 而不是 i128？因为 i64 的范围除以 10^8 还有约 922 亿，作为 USDT 计价完全够用。i128 在当前硬件上某些运算会慢一些（不是原生 64 位宽度），没必要为了用不到的范围付性能税。

基础运算

加减法最简单，raw 值直接加减：

impl Add for FixedPoint {
    type Output = Self;
    fn add(self, rhs: Self) -> Self {
        Self { raw: self.raw + rhs.raw }
    }
}

impl Sub for FixedPoint {
    type Output = Self;
    fn sub(self, rhs: Self) -> Self {
        Self { raw: self.raw - rhs.raw }
    }
}

乘法有个坑。两个 raw 值相乘，结果需要除以 SCALE 才能回到正确的定点表示。但 i64 × i64 可能溢出——50000 × 100 的 raw 值相乘就会超过 i64 范围。这里必须用 i128 做中间值：

impl Mul for FixedPoint {
    type Output = Self;
    fn mul(self, rhs: Self) -> Self {
        let result = (self.raw as i128 * rhs.raw as i128) / SCALE as i128;
        Self { raw: result as i64 }
    }
}

实测不走 i128 会怎样：

50000 × 100 用 i128: 5000000.00000000 ✓
直接 i64 相乘:       -9123183826594430976 （溢出）

除法反过来，先乘 SCALE 再除：

impl Div for FixedPoint {
    type Output = Self;
    fn div(self, rhs: Self) -> Self {
        assert!(rhs.raw != 0, "division by zero");
        let result = (self.raw as i128 * SCALE as i128) / rhs.raw as i128;
        Self { raw: result as i64 }
    }
}

构造和显示

从 f64 构造只在初始化时用，热路径上不应该出现浮点转换：

impl FixedPoint {
    const ZERO: Self = Self { raw: 0 };

    fn from_f64(v: f64) -> Self {
        Self { raw: (v * SCALE_F64).round() as i64 }
    }

    fn from_raw(raw: i64) -> Self {
        Self { raw }
    }

    fn to_f64(self) -> f64 {
        self.raw as f64 / SCALE_F64
    }
}

Display 实现要注意负数处理和前导零补齐：

impl fmt::Display for FixedPoint {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        let sign = if self.raw < 0 { "-" } else { "" };
        let abs_raw = self.raw.unsigned_abs();
        let integer = abs_raw / SCALE as u64;
        let frac = abs_raw % SCALE as u64;
        write!(f, "{}{}.{:08}", sign, integer, frac)
    }
}

{:08} 确保小数部分始终 8 位，0.00000001 不会被截成 0.1。

实际用途：持仓跟踪

定义一个 Position 结构来跟踪持仓、均价和盈亏：

struct Position {
    qty: FixedPoint,
    avg_cost: FixedPoint,
    realized_pnl: FixedPoint,
}

impl Position {
    fn on_fill(&mut self, fill_qty: FixedPoint, fill_price: FixedPoint) {
        if fill_qty.is_positive() {
            let old_cost = self.qty * self.avg_cost;
            let new_cost = fill_qty * fill_price;
            self.qty = self.qty + fill_qty;
            if self.qty.raw != 0 {
                self.avg_cost = (old_cost + new_cost) / self.qty;
            }
        } else {
            let sold_qty = fill_qty.abs();
            let pnl = sold_qty * (fill_price - self.avg_cost);
            self.realized_pnl = self.realized_pnl + pnl;
            self.qty = self.qty + fill_qty;
        }
    }

    fn unrealized_pnl(&self, market_price: FixedPoint) -> FixedPoint {
        self.qty * (market_price - self.avg_cost)
    }
}

每一步的均价和盈亏都是精确的整数运算，不存在浮点漂移。这组交易如果用 f64 算，单步结果看着一样，但把几千组这样的交易跑完再汇总，差异就会冒出来。

表示范围和溢出

i64 最大值 9,223,372,036,854,775,807 除以 10^8 约等于 922 亿。做加密货币交易，就算你管着一个中等规模基金的资金池，也很难碰到这个上限。

真正要小心的是乘法溢出。前面提到了 i128 中间值方案，但如果两个操作数本身就接近 i64 上限（比如 900 亿 × 900 亿），i128 也会溢出。实际业务中不太可能出现——BTC 总市值也就那个量级，单个持仓不可能到 900 亿 USDT。不过做防御性编程的话，可以加一个 checked 版本：

impl FixedPoint {
    fn checked_mul(self, rhs: Self) -> Option<Self> {
        let result = (self.raw as i128).checked_mul(rhs.raw as i128)?;
        let divided = result / SCALE as i128;
        if divided > i64::MAX as i128 || divided < i64::MIN as i128 {
            return None;
        }
        Some(Self { raw: divided as i64 })
    }
}

Benchmark

测试环境：Apple M2 Max, macOS 26.3.1, rustc 1.94.0, cargo bench (criterion 0.5)。

单次运算：

操作FixedPointf64
加法1.24 ns0.96 ns
乘法3.44 ns0.89 ns

单次运算 f64 更快，这符合预期——浮点加法是单条指令，定点乘法多了一次 i128 除法。

但看累加场景（1000 次连续加法）：

操作FixedPointf64
累加 1000 次294 ns864 ns

定点数快了约 3 倍。整数加法的数据依赖链比浮点短，CPU 流水线能更好地利用。实际交易系统里手续费、资金费率都是这种累加模式，定点数反而有优势。

为什么不用 rust_decimal 或 fixed crate

社区有现成方案：

  • rust_decimal 用 128 位存储，精度 28-29 位小数，主打金融场景。但我的需求固定是 8 位小数，128 位对 BTC 交易来说过重了，内存占用翻倍。
  • fixed crate 基于类型级别的泛型定义精度，灵活但编译时约束多，对我的场景来说抽象层太厚。

自己写的好处：完全控制精度和行为，性能摸得到底，代码量也就几十行。交易系统不适合用”通用解”——你需要知道每个 bit 在干什么。

几个踩过的坑

from_f64 的 round 不能省。 最初我用 as i64 直接截断，0.1 * 10^8 在 f64 里是 9999999.999…，截断后变成 9999999 而不是 10000000。加 .round() 才对。

Display 的 {:08} 不能写成 {:8}。 后者是右对齐填空格，0.00000001 会显示成 0. 1。

乘法的除法顺序不能反。 有人觉得先除 SCALE 再乘可以避免溢出，但那会丢精度。比如 raw=150000001（1.50000001）先除 10^8 变成 1（截断了），再乘就全错。必须先乘后除，靠 i128 撑住中间值。
