Rust 策略引擎多态之争：enum dispatch 并不总是比 dyn Trait 快

⏱ 2026年3月26日
· 32 分钟阅读
#Rust#性能优化#交易系统#benchmark

写交易策略引擎的时候，绕不开一个架构问题：策略的多态用什么方式实现？

Rust 里常见两种路子——dyn Trait（trait object，动态分发）和 enum + match（枚举分发）。社区里有一种广泛流传的说法：“enum dispatch 比 dyn dispatch 快，因为省掉了 vtable 间接跳转”。我之前也是这么认为的，直到上周把策略引擎从 dyn 切到 enum，跑了一次 benchmark，发现事情没那么简单。

问题场景

我手上的策略引擎需要同时运行多个策略：动量策略、均值回归策略、价差策略。每个 tick 进来，都要依次调用每个策略的 on_tick 方法，拿回信号。

核心循环大概长这样：

for tick in &ticks {
    for strategy in &mut strategies {
        match strategy.on_tick(tick) {
            Signal::Buy(_) | Signal::Sell(_) => signal_count += 1,
            Signal::Hold => {}
        }
    }
}

策略的数量在运行时确定（配置文件指定），类型也不固定。经典的多态场景。

dyn Trait 版本

先看 trait object 的写法，这是最直观的：

#[derive(Debug, Clone, Copy)]
pub struct MarketTick {
    pub price: f64,
    pub volume: f64,
    pub bid: f64,
    pub ask: f64,
}

#[derive(Debug, Clone, Copy)]
pub enum Signal {
    Buy(f64),
    Sell(f64),
    Hold,
}

pub trait Strategy {
    fn name(&self) -> &str;
    fn on_tick(&mut self, tick: &MarketTick) -> Signal;
}

动量策略的实现——跟踪一个滑动窗口，算区间收益率：

pub struct MomentumStrategy {
    window: Vec<f64>,
    capacity: usize,
    threshold: f64,
}

impl MomentumStrategy {
    pub fn new(capacity: usize, threshold: f64) -> Self {
        Self {
            window: Vec::with_capacity(capacity),
            capacity,
            threshold,
        }
    }
}

impl Strategy for MomentumStrategy {
    fn name(&self) -> &str { "momentum" }

    fn on_tick(&mut self, tick: &MarketTick) -> Signal {
        self.window.push(tick.price);
        if self.window.len() > self.capacity {
            self.window.remove(0);
        }
        if self.window.len() < self.capacity {
            return Signal::Hold;
        }
        let first = self.window[0];
        let last = self.window[self.capacity - 1];
        let ret = (last - first) / first;
        if ret > self.threshold {
            Signal::Buy(last)
        } else if ret < -self.threshold {
            Signal::Sell(last)
        } else {
            Signal::Hold
        }
    }
}

均值回归策略，算 z-score：

pub struct MeanReversionStrategy {
    window: Vec<f64>,
    capacity: usize,
    std_multiplier: f64,
}

impl MeanReversionStrategy {
    pub fn new(capacity: usize, std_multiplier: f64) -> Self {
        Self {
            window: Vec::with_capacity(capacity),
            capacity,
            std_multiplier,
        }
    }
}

impl Strategy for MeanReversionStrategy {
    fn name(&self) -> &str { "mean_reversion" }

    fn on_tick(&mut self, tick: &MarketTick) -> Signal {
        self.window.push(tick.price);
        if self.window.len() > self.capacity {
            self.window.remove(0);
        }
        if self.window.len() < self.capacity {
            return Signal::Hold;
        }
        let mean: f64 = self.window.iter().sum::<f64>() / self.capacity as f64;
        let var: f64 = self.window
            .iter()
            .map(|x| (x - mean).powi(2))
            .sum::<f64>() / self.capacity as f64;
        let std = var.sqrt();
        let z = (tick.price - mean) / std;
        if z < -self.std_multiplier {
            Signal::Buy(tick.price)
        } else if z > self.std_multiplier {
            Signal::Sell(tick.price)
        } else {
            Signal::Hold
        }
    }
}

再来一个简单的价差策略：

pub struct SpreadStrategy {
    threshold: f64,
}

impl SpreadStrategy {
    pub fn new(threshold: f64) -> Self {
        Self { threshold }
    }
}

impl Strategy for SpreadStrategy {
    fn name(&self) -> &str { "spread" }

    fn on_tick(&mut self, tick: &MarketTick) -> Signal {
        let spread = (tick.ask - tick.bid) / tick.price;
        if spread > self.threshold {
            Signal::Sell(tick.ask)
        } else if spread < self.threshold * 0.3 {
            Signal::Buy(tick.bid)
        } else {
            Signal::Hold
        }
    }
}

用的时候：

let mut strategies: Vec<Box<dyn Strategy>> = vec![
    Box::new(MomentumStrategy::new(20, 0.02)),
    Box::new(MeanReversionStrategy::new(50, 2.0)),
    Box::new(SpreadStrategy::new(0.001)),
];

enum dispatch 版本

把三个策略塞进一个枚举：

pub struct MomentumState {
    pub window: Vec<f64>,
    pub capacity: usize,
    pub threshold: f64,
}

pub struct MeanReversionState {
    pub window: Vec<f64>,
    pub capacity: usize,
    pub std_multiplier: f64,
}

pub struct SpreadState {
    pub threshold: f64,
}

pub enum StrategyEnum {
    Momentum(MomentumState),
    MeanReversion(MeanReversionState),
    Spread(SpreadState),
}

然后在 impl StrategyEnum 里写 on_tick，每个 arm 里放对应逻辑：

impl StrategyEnum {
    pub fn on_tick(&mut self, tick: &MarketTick) -> Signal {
        match self {
            StrategyEnum::Momentum(s) => {
                s.window.push(tick.price);
                if s.window.len() > s.capacity {
                    s.window.remove(0);
                }
                if s.window.len() < s.capacity {
                    return Signal::Hold;
                }
                let first = s.window[0];
                let last = s.window[s.capacity - 1];
                let ret = (last - first) / first;
                if ret > s.threshold {
                    Signal::Buy(last)
                } else if ret < -s.threshold {
                    Signal::Sell(last)
                } else {
                    Signal::Hold
                }
            }
            StrategyEnum::MeanReversion(s) => {
                s.window.push(tick.price);
                if s.window.len() > s.capacity {
                    s.window.remove(0);
                }
                if s.window.len() < s.capacity {
                    return Signal::Hold;
                }
                let mean: f64 =
                    s.window.iter().sum::<f64>() / s.capacity as f64;
                let var: f64 = s.window
                    .iter()
                    .map(|x| (x - mean).powi(2))
                    .sum::<f64>() / s.capacity as f64;
                let std = var.sqrt();
                let z = (tick.price - mean) / std;
                if z < -s.std_multiplier {
                    Signal::Buy(tick.price)
                } else if z > s.std_multiplier {
                    Signal::Sell(tick.price)
                } else {
                    Signal::Hold
                }
            }
            StrategyEnum::Spread(s) => {
                let spread = (tick.ask - tick.bid) / tick.price;
                if spread > s.threshold {
                    Signal::Sell(tick.ask)
                } else if spread < s.threshold * 0.3 {
                    Signal::Buy(tick.bid)
                } else {
                    Signal::Hold
                }
            }
        }
    }
}

Benchmark

用 criterion 跑 10 万个 tick，3 个策略并行处理：

use criterion::{Criterion, black_box, criterion_group, criterion_main};

fn bench_dyn_dispatch(c: &mut Criterion) {
    let ticks = generate_ticks(100_000);
    c.bench_function("dyn_dispatch_100k", |b| {
        b.iter(|| {
            let mut strategies: Vec<Box<dyn Strategy>> = vec![
                Box::new(MomentumStrategy::new(20, 0.02)),
                Box::new(MeanReversionStrategy::new(50, 2.0)),
                Box::new(SpreadStrategy::new(0.001)),
            ];
            black_box(run_dyn_dispatch(&mut strategies, &ticks))
        });
    });
}

fn bench_enum_dispatch(c: &mut Criterion) {
    let ticks = generate_ticks(100_000);
    c.bench_function("enum_dispatch_100k", |b| {
        b.iter(|| {
            let mut strategies = vec![
                StrategyEnum::Momentum(MomentumState {
                    window: Vec::with_capacity(20),
                    capacity: 20,
                    threshold: 0.02,
                }),
                StrategyEnum::MeanReversion(MeanReversionState {
                    window: Vec::with_capacity(50),
                    capacity: 50,
                    std_multiplier: 2.0,
                }),
                StrategyEnum::Spread(SpreadState { threshold: 0.001 }),
            ];
            black_box(run_enum_dispatch(&mut strategies, &ticks))
        });
    });
}

测试环境：

  • Mac Studio, Apple M2 Max, 32 GB
  • macOS 26.3.1
  • rustc 1.94.0 (4a4ef493e 2026-03-02)
  • cargo 1.94.0 (85eff7c80 2026-01-15)

结果：

dyn_dispatch_100k       time:   [6.6576 ms 6.6681 ms 6.6781 ms]
enum_dispatch_100k      time:   [8.7310 ms 9.2297 ms 9.7613 ms]

enum dispatch 反而慢了约 38%。而且注意 enum 版本的方差大得多（8.73ms ~ 9.76ms），稳定性也不如 dyn。

到底怎么回事

跑出这个结果的时候我一度怀疑 benchmark 写错了，又检查了一遍确认两边逻辑完全一致，信号输出数量也一样（都是 121487 个信号）。

想了一下，原因大概有这几个：

1. 函数体膨胀导致 i-cache 压力增大

dyn dispatch 的 on_tick 是三个独立函数，每个很短。CPU 执行到 vtable 间接跳转时，分支预测器（BTB）只需要记住 3 个目标地址，对于现代 CPU 来说根本不是问题。

enum dispatch 的 on_tick 是一个巨大的 match，三个 arm 的代码全部内联在同一个函数体里。编译器生成的机器码更长，热循环的指令缓存占用更大。M2 Max 的 L1i 是 192KB，不至于放不下，但代码局部性确实变差了。

2. 枚举大小 vs 胖指针大小

Box<dyn Strategy> 是个胖指针，16 字节。

StrategyEnum 的大小取决于最大的 variant。MomentumState 和 MeanReversionState 内部都有 Vec<f64>（24 字节）+ usize（8 字节）+ f64（8 字节），加上 discriminant，整个 enum 大约 48 字节。Vec<StrategyEnum> 的每个元素比 Vec<Box<dyn Strategy>> 大 3 倍。

在只有 3 个元素的时候这不是关键因素。但如果策略数量上去（比如 100 个），数组遍历时的 cache line 占用差异会更明显。

3. 分支预测在这个场景下很高效

循环里的策略顺序是固定的：永远是 Momentum → MeanReversion → Spread。这种重复模式对分支预测器来说太友好了。vtable 间接调用走的是 BTB（Branch Target Buffer），三个固定目标，几乎不会 miss。

所以 “省掉 vtable 间接跳转” 带来的理论收益，在实际硬件上被分支预测器完全吃掉了。

什么时候 enum dispatch 确实更快

不是说 enum dispatch 没用。它的优势场景是：

  • variant 很小且统一：比如一个 enum Op { Add, Sub, Mul, Div }，每个 arm 就几条指令，整个 match 编译出来比函数调用还短。
  • variant 数量很少（2-3 个）且逻辑简单：编译器可能直接把 match 优化成 cmov 或者分支合并。
  • 编译器能做更多优化：enum 的所有类型在编译期可见，理论上编译器可以做内联、常量传播等。但如果每个 arm 的代码足够复杂，这些优化的空间就很有限。
  • 需要避免堆分配：trait object 通常需要 Box，enum 可以放在栈上。在频繁创建销毁的场景下，省掉 allocation 可能比 dispatch 方式本身更重要。

实际项目里怎么选

我后来还是把策略引擎切回了 dyn Trait，原因不全是性能：

dyn Trait 的好处：

  • 可以在运行时动态加载策略（通过配置文件、甚至热加载 .so）
  • 每个策略是独立的类型，测试方便
  • 新增策略不用改已有代码，符合开闭原则
  • 编译速度更快（改一个策略不会导致引擎主模块重编）

enum dispatch 的好处：

  • 所有类型编译期可见，match 没覆盖到的 variant 编译器会报错
  • 不需要 Box，可以放在栈上或者嵌入其他结构体
  • 序列化/反序列化更直接（serde 对 enum 支持很好）

在交易策略引擎这个场景下，策略的 on_tick 逻辑本身就是计算密集的，dispatch 方式带来的开销相对于策略逻辑本身微乎其微。选 dyn 还是 enum，应该看代码组织和可维护性，而不是纠结那点 dispatch 开销。

一个教训

“X 比 Y 快”这种说法，脱离了具体场景就是空话。enum dispatch 省了一次间接跳转？没错。但间接跳转的成本取决于分支预测命中率，而分支预测命中率取决于调用模式。在策略引擎这种固定顺序、高重复的场景下，vtable 跳转几乎零成本。

反过来，enum dispatch 把所有逻辑塞进一个函数体，增加了指令缓存压力。在策略逻辑较复杂的情况下，这个负面影响反而更大。

use rand::Rng;

pub fn generate_ticks(n: usize) -> Vec<MarketTick> {
    let mut rng = rand::rng();
    let mut price = 100.0_f64;
    (0..n)
        .map(|_| {
            price += rng.random_range(-0.5..0.5);
            price = price.max(1.0);
            let spread = rng.random_range(0.01..0.1);
            MarketTick {
                price,
                volume: rng.random_range(100.0..10000.0),
                bid: price - spread / 2.0,
                ask: price + spread / 2.0,
            }
        })
        .collect()
}

pub fn run_dyn_dispatch(
    strategies: &mut [Box<dyn Strategy>],
    ticks: &[MarketTick],
) -> usize {
    let mut signal_count = 0usize;
    for tick in ticks {
        for s in strategies.iter_mut() {
            match s.on_tick(tick) {
                Signal::Buy(_) | Signal::Sell(_) => signal_count += 1,
                Signal::Hold => {}
            }
        }
    }
    signal_count
}

pub fn run_enum_dispatch(
    strategies: &mut [StrategyEnum],
    ticks: &[MarketTick],
) -> usize {
    let mut signal_count = 0usize;
    for tick in ticks {
        for s in strategies.iter_mut() {
            match s.on_tick(tick) {
                Signal::Buy(_) | Signal::Sell(_) => signal_count += 1,
                Signal::Hold => {}
            }
        }
    }
    signal_count
}
