rkyv 0.8 升级踩坑：用零拷贝序列化处理交易行情数据

⏱ 2026年3月22日
· 17 分钟阅读
#rust#serialization#trading#rkyv

最近把自己写的行情存储服务从 rkyv 0.7 升到 0.8，改动比预期大不少。这篇把迁移过程记下来，也顺带解释一下为什么在交易系统里用 rkyv 而不是直接 serde + bincode。

为什么不用 serde

不是说 serde 不好。大部分场景 serde 完全够用，序列化速度也快。问题在于行情数据的访问模式比较特殊。

交易系统收到 WebSocket 行情快照之后，大多数时候只需要读几个字段：bid 价格、ask 价格、当前成交量。然后决定要不要下单，或者更新策略状态。整个结构体里可能有二十几个字段，但实际访问的只有三四个。

serde 反序列化时必须把所有字段拷贝到新分配的内存里，不管你用没用。rkyv 做的事不一样：把 Rust 结构体按内存布局序列化，读取时直接把原始字节 cast 成 Rust 引用，不做内存分配，不做拷贝。你想读哪个字段，直接通过引用访问就行，其他字段的字节原地不动。

这个模式叫零拷贝反序列化（zero-copy deserialization）。FlatBuffers、Cap’n Proto 也是类似思路，只是 rkyv 的 API 是直接基于 Rust 的 derive macro，用起来比 schema 文件那套省事不少。

用在什么地方比较合适：把行情落盘到 mmap 文件，或者多进程通过共享内存传行情。这两个场景下，读取方直接指针访问，没有任何额外开销。如果你的行情服务用的是共享内存 IPC，rkyv 是一个自然的选择。

什么场景不适合：对外 API、日志、配置文件——这些还是 serde + JSON 或 TOML。rkyv 的格式不是自描述的，换个语言读就麻烦了，调试时用 hex dump 看数据也很费劲。

rkyv 0.8 的主要变化

从 0.7 到 0.8，API 层面有几处破坏性改动，迁移时一个一个踩过去的。

feature flag 变了

0.7 里用 derive 宏需要 features = ["derive"]，0.8 把 derive 宏内置了，不需要单独开启。想要序列化时的字节校验功能，启用 bytecheck：

[dependencies]
# rkyv 0.7（旧）
rkyv = { version = "0.7", features = ["derive"] }

# rkyv 0.8（新）
rkyv = { version = "0.8", features = ["bytecheck"] }

错误类型换成了 rancor

0.7 里 serialize/deserialize 用的是 rkyv 自己维护的错误类型，0.8 引入了 rancor crate 来统一处理错误。一般直接用 rkyv::rancor::Error 就好：

use rkyv::rancor::Error;

// 序列化
let bytes = rkyv::to_bytes::<Error>(&my_struct).unwrap();

// 反序列化
let back: MyStruct = rkyv::from_bytes::<MyStruct, Error>(&bytes).unwrap();

rancor 的设计目的是让调用方能选择错误处理策略：用 Error 是标准的 boxed error，用 Panic 是直接 panic（适合确定数据合法的场景，省去错误处理开销）。

零拷贝访问 API

0.7 里零拷贝访问用的是 archived_root，需要 unsafe：

// 0.7 写法（需要 unsafe）
let archived = unsafe { rkyv::archived_root::<MyStruct>(&bytes) };

0.8 改成了 rkyv::access，通过 bytecheck 在访问前做布局校验，不需要 unsafe：

// 0.8 写法（safe）
let archived = rkyv::access::<rkyv::Archived<MarketSnapshot>, Error>(&bytes).unwrap();

这个改动对我来说是好事。之前那个 unsafe 块放在 hot path 上挺别扭的，现在去掉了。

archived 字段类型是端序类型

这个是最让人困惑的变化。访问 archived 结构体的字段时，f64 字段不是 f64，而是 rend::f64_le（小端字节序的 f64 封装）。同样，u64 字段是 u64_le，u32 是 u32_le。

读取时需要 .into() 转换：

let archived = rkyv::access::<rkyv::Archived<MarketSnapshot>, Error>(&bytes).unwrap();

// archived.bid 的类型是 rend::f64_le，不是 f64
let bid: f64 = archived.bid.into();
let ts: u64 = archived.timestamp_us.into();

编译器报错会说 expected f64, found f64_le，按提示加 .into() 就行。这个设计是刻意的——rkyv 希望让端序转换显式化，在大小端混合的系统里能避免一些隐藏的 bug。

u8 类型不受影响，单字节没有端序问题。

实际代码

行情快照和订单结构体加上 rkyv derive：

use rkyv::{Archive, Deserialize, Serialize};
use rkyv::rancor::Error;

#[derive(Archive, Serialize, Deserialize, Debug, PartialEq, Clone)]
pub struct MarketSnapshot {
    pub symbol_id: u32,
    pub timestamp_us: u64,
    pub bid: f64,
    pub ask: f64,
    pub bid_size: f64,
    pub ask_size: f64,
    pub last_price: f64,
    pub volume: f64,
}

#[derive(Archive, Serialize, Deserialize, Debug, PartialEq, Clone)]
pub struct Order {
    pub order_id: u64,
    pub symbol_id: u32,
    pub price: f64,
    pub quantity: f64,
    pub side: u8,        // 0 = buy, 1 = sell
    pub order_type: u8,  // 0 = limit, 1 = market
    pub timestamp_us: u64,
}

序列化和零拷贝访问：

pub fn serialize_snapshot(snap: &MarketSnapshot) -> Vec<u8> {
    rkyv::to_bytes::<Error>(snap).unwrap().to_vec()
}

/// 零拷贝读取 bid 价格，不分配内存
pub fn read_bid_price(bytes: &[u8]) -> f64 {
    let archived = rkyv::access::<rkyv::Archived<MarketSnapshot>, Error>(bytes).unwrap();
    archived.bid.into()  // f64_le -> f64
}

/// 完整反序列化，需要修改数据时才用
pub fn deserialize_snapshot(bytes: &[u8]) -> MarketSnapshot {
    rkyv::from_bytes::<MarketSnapshot, Error>(bytes).unwrap()
}

测试（cargo test 全部通过，cargo clippy -- -D warnings 零警告）：

#[cfg(test)]
mod tests {
    use super::*;

    fn sample_snap() -> MarketSnapshot {
        MarketSnapshot {
            symbol_id: 1,
            timestamp_us: 1_741_600_000_000_000_u64,
            bid: 43250.75,
            ask: 43251.25,
            bid_size: 1.5,
            ask_size: 0.8,
            last_price: 43250.0,
            volume: 12345.67,
        }
    }

    #[test]
    fn test_roundtrip() {
        let snap = sample_snap();
        let bytes = serialize_snapshot(&snap);
        let back = deserialize_snapshot(&bytes);
        assert_eq!(snap, back);
    }

    #[test]
    fn test_zero_copy_bid() {
        let snap = sample_snap();
        let bytes = serialize_snapshot(&snap);
        let bid = read_bid_price(&bytes);
        assert!((bid - 43250.75).abs() < 1e-10);
    }

    #[test]
    fn test_order_roundtrip() {
        let order = Order {
            order_id: 9999,
            symbol_id: 1,
            price: 43200.0,
            quantity: 0.5,
            side: 0,
            order_type: 0,
            timestamp_us: 1_741_600_000_001_000_u64,
        };
        let bytes = rkyv::to_bytes::<Error>(&order).unwrap();
        let back: Order = rkyv::from_bytes::<Order, Error>(&bytes).unwrap();
        assert_eq!(order, back);
    }
}

序列化后的字节大小是固定的——MarketSnapshot 的字节数每次都一样，这正是 rkyv 内存布局语义的体现。跑 cargo run 可以看到实际输出：

Serialized 64 bytes
Zero-copy bid: 43250.75
Deserialized: bid=43250.75, ask=43251.25

64 字节 = 4 (u32) + 4 (padding) + 8 (u64) + 6×8 (f64) = 64。symbol_id 是 u32，后面跟 u64，中间有 4 字节对齐 padding。

关于 bytecheck 和 unsafe 权衡

0.8 引入 bytecheck 之前，rkyv 的零拷贝访问是 unsafe 的——调用方要自己保证字节数据是合法的。0.8 里 rkyv::access 在返回引用之前会做一次布局校验，校验通过才 ok。

如果你非常确定数据来源可信（比如本进程自己序列化的，立刻读回来），可以跳过校验：

// unsafe，跳过 bytecheck 校验
let archived = unsafe {
    rkyv::access_unchecked::<rkyv::Archived<MarketSnapshot>>(&bytes)
};

我在生产代码里没用 unsafe 版本。bytecheck 的开销是一次性的，发生在第一次 access 时，后续通过 archived 引用访问字段不会重复校验。省的那点开销换来 safe 代码，值得。

如果是共享内存 IPC，写入方和读取方是不同进程，数据来源不完全可控，更应该开启校验。

关于字符串和动态类型

上面的例子只用了基础类型（数值、整数），rkyv 也支持 String 和 Vec，但这两个类型序列化后的 archived 类型不同：

  • String → ArchivedString（包含指针偏移，指向后面的字节）
  • Vec<T> → ArchivedVec<T::Archived>

用起来类似，但要注意 archived 类型里的”指针”是相对偏移，不是绝对地址，所以 archived 对象不能独立于 bytes 缓冲区存活。这也是为什么 rkyv::access 返回的生命周期绑在 bytes 上。

对于行情数据，我倾向于用整数 ID 代替字符串（symbol_id: u32 而不是 symbol: String），一方面省序列化开销，另一方面也能避免动态类型带来的额外复杂度。

迁移小结

从 0.7 到 0.8 的迁移工作量不算太大。主要改动：

  1. 去掉 features = ["derive"]，加 features = ["bytecheck"]
  2. 把 rkyv::archived_root 换成 rkyv::access（顺便去掉 unsafe 块）
  3. 在访问 archived 字段的地方加 .into()
  4. 把 serialize/deserialize 的错误类型改成 rkyv::rancor::Error

编译器报错引导性挺好，按着改一遍基本就过了。f64_le 那个第一次见会愣一下，但其实加个 .into() 就搞定。

整个迁移在一个有几十个 struct 的项目里花了大概两小时，主要时间在跑测试确认行为一致。

项目用的 rkyv 0.8.15，rustc 1.94.0，macOS 15.7.3（Apple Silicon）。
