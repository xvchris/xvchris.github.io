Rust 1.94 实战升级：array_windows、Cargo config include 和 TOML 1.1

⏱ 2026年3月10日
· 12 分钟阅读
#Rust#版本更新#性能优化

Rust 1.94.0 上周刚发布，我花了一个周末把手上几个项目都升级了。这个版本没有什么惊天动地的语法变更，但有几个小改进用起来确实舒服——特别是 array_windows 和 Cargo config include，直接解决了我之前一直在绕弯的问题。

array_windows：终于不用手动索引了

写过滑动窗口相关逻辑的人都知道，之前 slice::windows() 返回的是 &[T]，每次都得手动索引或者 try_into 转数组。1.94 稳定了 array_windows，直接返回 &[T; N]，编译器帮你搞定长度。

之前我在交易系统里有一段检测价格异常波动的代码：

// 旧写法 — 丑陋且有运行时开销
fn detect_spike(prices: &[f64], threshold: f64) -> Vec<usize> {
    let mut spikes = Vec::new();
    for i in 0..prices.len().saturating_sub(2) {
        let prev = prices[i];
        let curr = prices[i + 1];
        let next = prices[i + 2];
        if (curr - prev).abs() > threshold && (curr - next).abs() > threshold {
            spikes.push(i + 1);
        }
    }
    spikes
}

手动索引，边界处理烦人，编译器也没法做好优化。换成 array_windows：

// Rust 1.94 — 清爽
fn detect_spike(prices: &[f64], threshold: f64) -> Vec<usize> {
    prices
        .array_windows()
        .enumerate()
        .filter_map(|(i, &[prev, curr, next])| {
            if (curr - prev).abs() > threshold && (curr - next).abs() > threshold {
                Some(i + 1)
            } else {
                None
            }
        })
        .collect()
}

模式匹配直接解构 [prev, curr, next]，编译器自动推断窗口大小是 3。没有运行时 bounds check，LLVM 能更好地向量化。

除了代码更清晰之外，少了 bounds check 也意味着编译器有更多优化空间——LLVM 能更激进地做循环向量化，尤其在大数据量场景下差异会更明显。

更妙的是检测模式匹配。之前同事写过一段识别 K 线形态的代码，检测「吞没形态」（前一根阴线被后一根阳线完全包裹）：

#[derive(Clone, Copy)]
struct Candle {
    open: f64,
    close: f64,
}

fn find_bullish_engulfing(candles: &[Candle]) -> Vec<usize> {
    candles
        .array_windows()
        .enumerate()
        .filter_map(|(i, [prev, curr])| {
            let prev_bearish = prev.close < prev.open;
            let curr_bullish = curr.close > curr.open;
            let engulfs = curr.open <= prev.close && curr.close >= prev.open;
            (prev_bearish && curr_bullish && engulfs).then_some(i + 1)
        })
        .collect()
}

两个元素的窗口，编译器推断 N=2，解构成 [prev, curr]——读起来就像自然语言描述。

Cargo config include：多环境配置不再拼拼凑凑

做链上相关项目经常需要在不同环境间切换：本地开发用 devnet，CI 上跑 testnet，生产环境用 mainnet。之前要么搞一堆 feature flag，要么在 build.rs 里读环境变量。

1.94 的 Cargo config include 直接解决这个问题。在 .cargo/config.toml 里：

include = [
    { path = "config-base.toml" },
    { path = "config-local.toml", optional = true },
]

[env]
RUST_LOG = "info"

config-base.toml 放到仓库里，所有人共享：

[env]
RPC_URL = "https://api.devnet.solana.com"
CHAIN_ID = "devnet"

[build]
rustflags = ["-C", "target-cpu=native"]

config-local.toml 加到 .gitignore，每个开发者自己维护：

[env]
RPC_URL = "http://localhost:8899"
PRIVATE_KEY_PATH = "/home/chris/.config/solana/id.json"

CI 环境再搞一个 config-ci.toml：

[env]
RPC_URL = "https://api.testnet.solana.com"
RUST_LOG = "warn"

关键是 optional = true——本地没有 config-local.toml 也不会报错，直接用 base 配置。

之前我们团队的做法是每个人复制一份 .cargo/config.toml.example 然后手动改，新人入职经常忘记改 RPC 地址就跑测试，链上直接发了笔垃圾交易到 devnet 上。现在 base 配置兜底，本地文件覆盖，干净多了。

TOML 1.1：多行 inline table

这个改动看起来小，体验提升巨大。之前 Cargo.toml 里稍微复杂点的依赖就得写成一行，看着难受：

# 旧写法 — 一行到头
tokio = { version = "1.41", features = ["rt-multi-thread", "macros", "net", "time", "sync", "signal"] }

现在可以换行了：

# TOML 1.1 — 终于能呼吸了
tokio = {
    version = "1.41",
    features = [
        "rt-multi-thread",
        "macros",
        "net",
        "time",
        "sync",
        "signal",
    ],
}

尾逗号也支持了。改 features 的时候 git diff 终于干净了——之前加一个 feature 会把整行都标红标绿，review 的时候很烦。

有一点要注意：用了 TOML 1.1 语法之后，你的开发 MSRV 会变成 1.94+，因为老版本 Cargo 解析不了。但 cargo publish 的时候会自动重写成兼容格式，所以你的 crate 用户不受影响。我的做法是内部项目直接用新语法，开源库暂时还是保持旧格式。

其他值得留意的

LazyCell/LazyLock 的 get 和 get_mut：之前只能 force() 或解引用来访问，现在可以 get() 拿 Option，不触发初始化。做配置懒加载的时候有用——比如「如果已经初始化了就用，没有就走默认值」。

use std::sync::LazyLock;

static CONFIG: LazyLock<String> = LazyLock::new(|| {
    std::fs::read_to_string("/etc/app.conf").unwrap_or_default()
});

fn get_config_if_ready() -> Option<&'static str> {
    // 不触发初始化，只看看有没有值
    // 注意：不能写 CONFIG.get()，会被 Deref 到 str::get() 遮蔽
    LazyLock::get(&CONFIG).map(|s| s.as_str())
}

Peekable::next_if_map：peek + map + advance 一步到位。写 tokenizer 或者手工 parser 的时候很好使。

std::f64::consts::GOLDEN_RATIO 和 EULER_GAMMA：数学常量进标准库了。之前每个项目自己 const GOLDEN_RATIO: f64 = 1.618033988749895;，现在直接 use std::f64::consts::GOLDEN_RATIO; 就行。

升级建议

rustup update stable
rustc --version  # 确认 1.94.0

如果你的项目用了 windows() 做固定长度滑动窗口，grep 一下全换成 array_windows()。改动量不大，收益明确。

Cargo config include 建议在团队项目里立刻用起来——尤其是有多环境配置需求的，比旧方案优雅太多。

TOML 1.1 的多行 inline table 随手就能改，改完 Cargo.toml 的可读性立竿见影。
