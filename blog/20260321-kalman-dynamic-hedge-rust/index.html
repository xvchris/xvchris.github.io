用卡尔曼滤波器追踪动态对冲比率：配对交易里最容易踩的一个坑

⏱ 2026年3月21日
· 21 分钟阅读
#rust#quantitative-finance#trading#kalman-filter#statistical-arbitrage

之前写过配对交易的 Rust 实现，用 Engle-Granger 两步法做协整检验，跑出来的对冲比率是固定的。这个假设很强：认为 Y 和 X 之间的关系在整个回测期间保持不变。

真实市场不是这样运作的。

在合成实验里模拟一个 beta 从 1.0 缓慢漂移到 1.5 的场景，就能直接看到问题：用早期历史数据估计的固定 beta 计算出来的 spread 不再是平稳序列，z-score 信号几乎全是假信号。加密货币这类波动剧烈的市场里，beta 漂移更快、幅度更大，这个问题尤为明显。

解决方案是卡尔曼滤波器。让 beta 成为一个可以随时间演化的状态变量，用每一条新数据更新估计。

问题在哪里

配对交易的基本假设是：

price_Y = β × price_X + α + ε

OLS 的做法是用历史数据估计 β 和 α，然后固定下来。spread = price_Y - β × price_X - α，然后对 spread 做 z-score，偏离太大就开仓。

问题是 β 本身会漂移。市场结构改变、资金流向变化、相关性破裂，都会让这个比率缓慢偏移。你用半年前算的 β 来算今天的 spread，相当于在用一把已经偏了的尺子量东西。

有人用滚动窗口 OLS 来部分解决这个问题——每隔一段时间重新回归。但这里有个时间窗口选择的困境：窗口太短，估计不稳定；窗口太长，反应太慢。

卡尔曼滤波器换了一种思路：把 β 和 α 当成状态变量，每观测到一个新价格就做一次贝叶斯更新。不需要选窗口长度，而是通过一个参数 δ（过程噪声）来控制参数变化的速度。

状态空间建模

卡尔曼滤波器需要把问题写成状态空间形式：

状态方程：x_t = F × x_{t-1} + w_t    (w ~ N(0, Q))
观测方程：z_t = H_t × x_t + v_t       (v ~ N(0, R))

对应到配对交易：

  • 状态 x_t = [β_t, α_t]，两个时变参数
  • 状态转移矩阵 F = I（单位矩阵），假设参数做随机游走
  • 观测矩阵 H_t = [price_X_t, 1]，随时间变化
  • 观测值 z_t = price_Y_t
  • 过程噪声 Q = δ × I，控制参数漂移速度
  • 观测噪声 R，价格序列的残差方差

随机游走假设（F = I）是这里的关键。它的含义是：下一时刻的 beta 约等于当前时刻的 beta，加上一个小扰动。没有强制的均值回归，也没有趋势。这对很多金融参数是合理的先验。

Rust 实现

不依赖任何外部库，纯标准库实现。矩阵运算全部手写，2×2 的情况完全可以内联展开。

#[derive(Debug, Clone)]
pub struct KalmanFilter {
    /// 状态估计 [beta, alpha]
    pub state: [f64; 2],
    /// 状态协方差矩阵（2×2，行优先存储）
    pub cov: [[f64; 2]; 2],
    /// 过程噪声协方差
    pub q: [[f64; 2]; 2],
    /// 观测噪声方差
    pub r: f64,
}

impl KalmanFilter {
    /// delta: 过程噪声大小。越大表示参数变化越快。
    /// 典型范围：1e-5（慢变）到 1e-3（快变）
    pub fn new(delta: f64, r: f64) -> Self {
        // 初始协方差设很大：对 beta/alpha 的初始值完全没有信心
        let p_init = 1e4;
        KalmanFilter {
            state: [0.0, 0.0],
            cov: [[p_init, 0.0], [0.0, p_init]],
            q: [[delta, 0.0], [0.0, delta]],
            r,
        }
    }

    /// 输入一组新的 (y, x) 观测，返回更新后的 (beta, alpha, spread)
    pub fn update(&mut self, y: f64, x: f64) -> (f64, f64, f64) {
        // === 预测步 ===
        // F = I，所以状态预测就是当前状态
        // 但协方差要加上过程噪声：P_pred = P + Q
        let p_pred = [
            [self.cov[0][0] + self.q[0][0], self.cov[0][1] + self.q[0][1]],
            [self.cov[1][0] + self.q[1][0], self.cov[1][1] + self.q[1][1]],
        ];

        // 观测矩阵 H = [x, 1]
        let h = [x, 1.0];

        // 新息（innovation）：实际观测值 - 预测观测值
        let y_pred = h[0] * self.state[0] + h[1] * self.state[1];
        let innovation = y - y_pred;

        // 新息协方差 S = H P_pred H^T + R
        let hp = [
            h[0] * p_pred[0][0] + h[1] * p_pred[1][0],
            h[0] * p_pred[0][1] + h[1] * p_pred[1][1],
        ];
        let s = hp[0] * h[0] + hp[1] * h[1] + self.r;

        // 卡尔曼增益 K = P_pred H^T / S
        let k = [
            (p_pred[0][0] * h[0] + p_pred[0][1] * h[1]) / s,
            (p_pred[1][0] * h[0] + p_pred[1][1] * h[1]) / s,
        ];

        // === 更新步 ===
        // 状态更新：x = x_pred + K * innovation
        self.state[0] += k[0] * innovation;
        self.state[1] += k[1] * innovation;

        // 协方差更新：P = (I - KH) P_pred
        let i_kh = [
            [1.0 - k[0] * h[0], -k[0] * h[1]],
            [-k[1] * h[0], 1.0 - k[1] * h[1]],
        ];
        self.cov = [
            [
                i_kh[0][0] * p_pred[0][0] + i_kh[0][1] * p_pred[1][0],
                i_kh[0][0] * p_pred[0][1] + i_kh[0][1] * p_pred[1][1],
            ],
            [
                i_kh[1][0] * p_pred[0][0] + i_kh[1][1] * p_pred[1][0],
                i_kh[1][0] * p_pred[0][1] + i_kh[1][1] * p_pred[1][1],
            ],
        ];

        let spread = y - (self.state[0] * x + self.state[1]);
        (self.state[0], self.state[1], spread)
    }

    /// 当前 beta 估计的标准差（不确定性）
    pub fn hedge_ratio_std(&self) -> f64 {
        self.cov[0][0].sqrt()
    }
}

代码量不大，但里面有几个值得注意的地方。

初始协方差设大。p_init = 1e4 表示对初始 beta/alpha 完全没有先验。滤波器最初几个 tick 的更新会很大、很不稳定——这是正常的”预热”阶段，通常需要几十到一百个数据点才能收敛到合理值。实盘里一般会跳过前 100 个 tick 再开始使用信号。

K 是增益向量，不是矩阵。因为 H 是行向量（1×2），观测值是标量，所以卡尔曼增益 K 是列向量（2×1）。这让代码比一般教材里的矩阵形式简单很多。

spread 的计算。注意 spread 不是用预测状态算的，是用更新后的状态。这样 spread 包含了当前 tick 的信息，更灵敏。

delta 参数怎么选

这是整个方法最难的部分，没有放之四海而皆准的答案。

delta 控制参数随机游走的方差。小的 delta 让参数变化缓慢，平滑但反应迟钝；大的 delta 让参数变化快，灵敏但噪音大。

我自己的经验：

  • 加密货币：市场结构变化快，delta = 1e-4 到 1e-3 比较合适
  • 股票配对：结构相对稳定，delta = 1e-5 到 1e-4
  • 跨资产：差异大的资产对，可能需要 1e-3 甚至更高

一个检验方式是看 spread 序列的均值回归性：调节 delta 直到 spread 看起来最像平稳序列（ADF 检验 p 值最小，或者 half-life 最合理）。也可以用最大似然估计 delta 和 R，但计算量大很多。

还有一个实用判断：如果你发现用某个 delta 跑出来的 beta 总是滞后市场结构变化大约 N 个 bar，那就把 delta 调大一点。

z-score 信号生成

spread 出来之后，信号逻辑和静态配对交易一样，用滚动窗口算 z-score：

pub fn zscore(spread: &[f64], window: usize) -> Vec<f64> {
    spread
        .windows(window)
        .map(|w| {
            let mean = w.iter().sum::<f64>() / w.len() as f64;
            let var = w.iter().map(|x| (x - mean).powi(2)).sum::<f64>()
                / (w.len() - 1) as f64;
            let std = var.sqrt();
            if std < 1e-10 {
                0.0
            } else {
                (w[w.len() - 1] - mean) / std
            }
        })
        .collect()
}

用样本标准差（除以 n-1）而不是总体标准差（除以 n），在小窗口下更保守。1e-10 的保护是防止 spread 全是常数时的除零。

信号生成加了状态机，不允许重复开仓：

pub fn generate_signals(
    zscores: &[f64],
    entry_threshold: f64,
    exit_threshold: f64,
) -> Vec<Signal> {
    let mut signals = Vec::with_capacity(zscores.len());
    let mut in_position: Option<Signal> = None;

    for &z in zscores {
        let signal = match in_position {
            None => {
                if z > entry_threshold {
                    in_position = Some(Signal::Short);
                    Signal::Short
                } else if z < -entry_threshold {
                    in_position = Some(Signal::Long);
                    Signal::Long
                } else {
                    Signal::Hold
                }
            }
            Some(Signal::Short) => {
                if z < exit_threshold {
                    in_position = None;
                    Signal::Exit
                } else {
                    Signal::Short
                }
            }
            Some(Signal::Long) => {
                if z > -exit_threshold {
                    in_position = None;
                    Signal::Exit
                } else {
                    Signal::Long
                }
            }
            Some(_) => Signal::Hold,
        };
        signals.push(signal);
    }
    signals
}

进场阈值和出场阈值分开，避免 z-score 在临界点反复横跳触发频繁交易。我通常用 entry=2.0、exit=0.5，或者 entry=1.5、exit=0.0。

性能

测试环境：Mac Studio（Apple M2 Max，32 GB），macOS 15.7.3，rustc 1.94.0。

kalman_update_10k   time: [133.33 µs 133.56 µs 133.79 µs]

处理 10,000 个 tick 约 134µs，平均每次 update 13.4ns。实时行情流完全够用，即使是 tick 级别的加密货币数据（Binance 主流币对大约每秒 10-50 个 tick）。

不依赖 nalgebra 这类线性代数库，避免了泛型和动态分发的开销。2×2 矩阵运算在编译时完全展开，没有堆分配。

预热期问题

这个坑踩过一次。卡尔曼滤波器在最开始的数十到数百个 tick 里，状态估计不稳定，beta 会大幅跳动。这段时间不应该产生交易信号。

简单的处理方式：

let warmup = 100;
let mut kf = KalmanFilter::new(1e-4, 1.0);
let mut spreads = Vec::new();

for (i, (&y, &x)) in prices_y.iter().zip(prices_x.iter()).enumerate() {
    let (_, _, spread) = kf.update(y, x);
    if i >= warmup {
        spreads.push(spread);
    }
}

更精细的做法是看 kf.hedge_ratio_std()——当 beta 的标准差降到某个阈值以下再开始使用信号。这个值初始是 100（sqrt(1e4)），随着数据积累快速下降。

和静态 OLS 的对比

静态 OLS滚动窗口 OLS卡尔曼滤波器
适应市场结构变化✗部分（窗口内）✓
参数估计稳定性好取决于窗口好（可调）
计算复杂度一次性O(window)O(1) per tick
超参数无窗口大小delta, R
不确定性量化✗部分✓（协方差矩阵）

卡尔曼滤波器额外输出协方差矩阵，这意味着你能知道当前 beta 估计有多不可靠。在不确定性高的时候减小仓位或者暂停交易，是一个直接可用的风险控制思路。
